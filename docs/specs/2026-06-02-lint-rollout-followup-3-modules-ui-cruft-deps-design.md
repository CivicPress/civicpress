## 1. Goal

Remove ESLint-related dependencies from `modules/ui/package.json` that were left
over from the L1-T4 Option B → Option A detour during the lint-rule rollout.
Confirm lint continues to pass via the `@nuxt/eslint` Option A integration that
`modules/ui/eslint.config.mjs` actually uses today.

This is the smallest of the four 2d-followups and is intended as a single
self-contained session.

## 2. Scope

**In scope** — `modules/ui/package.json` only. Three candidate deps:

| Dep                                | Section         | Status                                                 |
| ---------------------------------- | --------------- | ------------------------------------------------------ |
| `@eslint/js@9.39.4`                | devDependencies | Safe leftover (no direct import in `modules/ui`)       |
| `vue-eslint-parser@10.4.0`         | devDependencies | Safe leftover (no direct import in `modules/ui`)       |
| `@typescript-eslint/parser@8.60.0` | devDependencies | Probe — may be transitively required by `@nuxt/eslint` |

**Kept** — `@typescript-eslint/eslint-plugin` (imported as `pluginTs` in
`modules/ui/eslint.config.mjs:3`).

**Out of scope:**

- Other workspaces' eslint configs (`core`, `cli`, `modules/api`,
  `modules/storage` all still reference `@eslint/js`/`vue-eslint-parser`
  legitimately — leave alone)
- Version bumps of any retained dep
- The other three 2d-followups (`unused-vars`, Vue-template `any` blind spot,
  `STYLE_RULES_DEFERRED`)
- Any change to `eslint.config.mjs` itself
- Any push to origin (refactor push policy: nothing until phase 7)

## 3. Verification of "leftover" claim

`grep -rn` across `modules/ui/**/*.{ts,vue,mjs,cjs,js,json}` excluding
`node_modules` and `.nuxt`:

- `@eslint/js` — only hit: `modules/ui/package.json:43`
- `vue-eslint-parser` — only hit: `modules/ui/package.json:56`
- `@typescript-eslint/parser` — only hit: `modules/ui/package.json:47`

The eslint config (`modules/ui/eslint.config.mjs`) imports exactly two things:

1. `withNuxt` from `./.nuxt/eslint.config.mjs` (regenerated by `nuxt prepare`)
2. `pluginTs` from `@typescript-eslint/eslint-plugin`

Whatever parser ESLint uses for `.vue` and `.ts` files flows through
`withNuxt(...)`; the `@nuxt/eslint` package declares its own parser deps.

## 4. Approach — two-step bisect

### Step A — combined safe removal

1. Edit `modules/ui/package.json`: delete the `@eslint/js` and
   `vue-eslint-parser` entries from `devDependencies`.
2. From repo root, run `pnpm install`.
3. Regenerate the Nuxt eslint config:
   `pnpm --filter @civicpress/ui exec nuxt prepare`.
4. Lint: `pnpm --filter @civicpress/ui exec eslint .` — must exit 0 (or match
   the current `dev`-branch baseline; existing warnings are acceptable, **no new
   errors**).
5. Commit on a new branch off `dev`: `refactor/lint-followup-3-ui-cruft-deps`.
   Conventional message:
   `refactor(lint-followup-3): drop @eslint/js + vue-eslint-parser from modules/ui`.

### Step B — probe `@typescript-eslint/parser`

1. Edit `modules/ui/package.json`: delete the `@typescript-eslint/parser` entry.
2. Repeat `pnpm install` → `nuxt prepare` → eslint run.
3. **If lint passes:** commit on the same branch. Message:
   `refactor(lint-followup-3): drop @typescript-eslint/parser from modules/ui (transitively provided)`.
4. **If lint fails:** `git restore modules/ui/package.json pnpm-lock.yaml`, then
   record the constraint in the closure followup memory (one sentence: "retained
   because `@nuxt/eslint` does not transitively expose
   `@typescript-eslint/parser` — removal breaks `<exact failure>`"). Do not
   force the removal. Package.json itself stays untouched for Step B; no
   comment-strings since JSON doesn't support comments.

## 5. Alternatives considered

| Alternative                               | Why not                                                                                                                                                                                      |
| ----------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| All-at-once: drop all 3 in one commit.    | Simpler diff, but if eslint breaks the bisect is manual; the parser-resolution ambiguity is the entire interesting part of this work, so we shouldn't fold it into a "safe-leftover" commit. |
| One-at-a-time (3 commits).                | Three installs / three lint runs for two deps that are equally clearly safe. Pedantic without payoff.                                                                                        |
| Defer until Step #1 (unused-vars) starts. | Slight integration win, but unused-vars is multi-day and the cruft removal is unrelated. Keeping followups independent matches the memory guidance ("don't try to fix all three together").  |

## 7. Risks & rollback

- **Lockfile churn beyond expected scope.** Mitigation: review `pnpm-lock.yaml`
  diff before committing. If it shows unrelated package movement, abort and
  investigate.
- **Lint config silently uses a removed parser.** Mitigation: `nuxt prepare`
  regenerates `.nuxt/eslint.config.mjs` from current installed packages — a
  missing parser surfaces as an install or lint error, not a silent regression.
- **Step B fails for a non-obvious reason.** Rollback path:
  `git restore modules/ui/package.json pnpm-lock.yaml` — single command, no
  follow-on cleanup.
