## Technical Quality

Code is well-typed end-to-end (strict TS, exhaustive types in `civic-core.ts`, `records/record-manager.ts`). Errors use a unified hierarchy (`errors/index.ts`, `errors/domain-errors.ts`) with `correlationId`, status codes, and `toJSON()` — a real improvement over ad-hoc throws. However, individual file sizes are large: `record-manager.ts` is 1,420 LoC, `database-service.ts` is 1,577 LoC, `auth-service.ts` is 1,319 LoC, `saga-executor.ts` is 534 LoC. These god-objects are concerning for v0.2 alpha and accumulate responsibilities (DB, file I/O, validation, hooks, audit, git, cache) inside a single class.

Test coverage is uneven. The DI container has 7 dedicated test files; the saga module has 6 integration tests including failure injection. By contrast, `core/src/audit/audit-logger.ts` (the platform's stated transparency feature) has no dedicated test, and the workflow engine's three "default workflows" (`approvalWorkflow`, `publicationWorkflow`, `archivalWorkflow`) are TODO stubs (`workflows/workflow-engine.ts:99-134`) yet are still registered as if functional. `core/src/hooks/hook-system.ts` has a `logHook` method (line 414) whose implementation comment reads `// TODO: Implement proper log writing` — the system claims to write `.civic/hooks.log.jsonl` but the file is never produced.

Documentation drift is significant. `core/README.md` documents a "RecordManager + HookSystem + WorkflowEngine + GitEngine" four-component architecture (lines 18-30). The actual code is at least double that: DI container, saga executor + state store + idempotency + resource lock + recovery + metrics, unified cache manager with three strategies, secrets manager (HKDF), notification subsystem (~2,465 LoC under `core/src/notifications/`), audit logger, indexing service, diagnostics service. The README has not been updated to reflect what was actually built; this is a Transparency-principle problem when the core module's own docs misrepresent the system.

14 `TODO` markers remain in production code (excluding tests). Some are benign (Postgres adapter not yet built), but `geography/geography-manager.ts` has four TODOs about not actually persisting to the database (lines 115, 292, 360, 605), which suggests the geography feature is half-wired.

## Security (LIGHT)

Auth foundations look sound. `AuthService` uses `crypto.randomBytes(32)` for tokens, sha256-hashes them in the DB, and optionally signs them with HKDF-derived keys from a root secret stored at `.system-data/secrets.yml` or `CIVICPRESS_SECRET` env var (`security/secrets.ts`). Bcrypt is used for passwords (12 rounds per project-status.md, not directly verified here). SQL is parameterised throughout `database-service.ts` and `saga-state-store.ts` (no string concatenation observed in queries inspected). CSRF protection is registered (`security/csrf.ts` exists).

The audit-trail design is the main civic security concern. `RecordManager.updateRecord` writes audit events to the `audit_logs` table without populating `user_id` (`record-manager.ts:778-785` — no `userId` field passed even though `user: AuthUser` is in scope), so the DB audit table for records doesn't record who did what. The saga update/create/archive paths (`saga/*-saga.ts`) write nothing to either audit channel — all auditing relies on API-route-level wrapping in `modules/api/src/routes/records.ts` and equivalent CLI wrappers. This makes the audit trail brittle: any code path that bypasses the API/CLI layer (background sync, internal call, future module) silently produces unaudited civic changes. For a civic platform whose Trust and Transparency principles depend on "who changed what, when", this is a structural gap, not a polish item.

Surfaces flagged for a dedicated security session: (1) full authn/authz endpoint matrix across `modules/api`; (2) JWT vs session-token split in `auth-service.ts` (two parallel token systems coexist — JWT for some flows, opaque tokens for sessions/api-keys); (3) the OAuth flow in `oauth-provider.ts` (state parameter handling, callback validation); (4) the secrets manager fallback to auto-generated dev secrets (`security/secrets.ts:71`) which should be impossible in production; (5) the `email-validation-service.ts` token flow.

## AI-Generation Smells

**Over-engineering at v0.2 alpha is the strongest signal.** Core ships a hand-rolled DI container (`di/` — 7 files, ~600 LoC, with circular-dep detector and dependency graph visualiser) for an application whose registration list is 11 services manually wired in `civic-core-services.ts:43-199`. The `DependencyResolver.extractDependencies()` method returns `[]` and has a TODO comment about future decorator support (`di/dependency-resolver.ts:57-64`); the container is effectively a service locator with extra ceremony. A `Map<string, factory>` would deliver the same functionality.

**Dead infrastructure built ahead of need.** `SagaRecovery` (`saga/saga-recovery.ts`, 234 LoC) and `SagaMetricsCollector` (`saga/saga-metrics.ts`) are exported from `saga/index.ts` but are **never instantiated by production code** — `grep` shows them used only in `saga-e2e.test.ts`. `CacheWarmer` (`cache/warming/cache-warmer.ts`, 203 LoC) is wired into `UnifiedCacheManager` but no `registerFromConfig` call in the codebase sets `warming.enabled: true`. The `'hybrid'` cache strategy in `CacheConfig` (`cache/types.ts:105`) is declared as a valid option but `unified-cache-manager.ts:73-77` throws `"Hybrid cache strategy not yet implemented"`. This is fake-comprehensiveness: typed options that don't exist.

**Fake comprehensiveness in workflows and hooks.** `WorkflowEngine` exposes `approvalWorkflow`, `publicationWorkflow`, `archivalWorkflow` (`workflows/workflow-engine.ts:99-134`) — all three are TODO bodies that only log "X workflow started". `HookSystem.logHook` (line 414) similarly stubs the actual log write. These are scaffolding-shaped methods that present the appearance of a workflow engine without delivering one.

**Two parallel audit systems.** `core/src/audit/audit-logger.ts` (JSONL file, used by API/CLI routes — 48 call sites) and `DatabaseService.logAuditEvent` (SQLite `audit_logs` table, used internally — only 1 call in `record-manager.ts`). They are not coordinated; the same civic event is sometimes logged in one, sometimes in the other, sometimes in neither. A `tail()`-based admin UI is built on top of the file logger; nothing reads the DB table outside the CLI's `civic status`.

**Notifications duplicated in core + modules/notifications.** `core/src/notifications/` contains ~2,465 LoC of channel, queue, template, security, rate-limiter, logger, audit code. There is also a separate `modules/notifications/` module. The intent is unclear from inside core.

**Comments that explain WHAT not WHY** appear throughout the saga module (every step has `// Step 1: ...`, `// Step 2: ...` block comments narrating the obvious). The saga step classes also include `coreDebug` of step start/end in addition to the executor's own logging, producing double-logging.

## Architecture

Module boundaries are mostly clean at the export level (`core/src/index.ts` is the single re-export hub). However, the layering inside core is **inverted in places**: `RecordManager` dynamically `import('../saga/index.js')` (`record-manager.ts:809-818`) to avoid a circular dependency with the saga module that it depends on. This is a code smell suggesting saga and record-manager are at the wrong layer: sagas should orchestrate record operations, not the other way around. `civic-core.ts:229-232` also uses dynamic `await import('./civic-core-services.js')` to dodge a likely circular import — a sign that the service-registration flow is fighting itself.

Integration with sub-systems is somewhat ad-hoc. The `IndexingService` is registered as a **placeholder object** at first (`civic-core-services.ts:147-151`) then replaced during `completeServiceInitialization()` (line 235), because `IndexingService` needs a `CivicPress` instance — circular by construction. The author left a comment "Note: This is a workaround — ideally IndexingService wouldn't need CivicPress" (line 233). The DI container's failure to express this dependency natively is a sign the DI design doesn't fit this codebase yet.

The storage module integration is unusually defensive: `civic-core-services.ts:244-349` is ~100 lines of nested try/catch that attempts package imports, then file:// URL imports, with multiple fallbacks and conditional logging. This pattern suggests that the workspace package resolution was fighting the codebase during build/test, and was solved by piling on fallbacks rather than fixing the build setup. It works but it's a fragile seam.

Saga design itself is **structurally correct for the civic problem**: the ordering (DB → file → Git commit → indexing/hooks) means a crash before commit leaves no persisted civic artefact, and the explicit "Git commits are never rolled back" rule (`saga-executor.ts:469-485`) matches the manifesto's archival constraint (you don't un-commit a public decision). Compensation logic exists and is tested in `saga-failure-injection.test.ts`. The concern is not the design but the unused recovery scaffolding around it.

## Roadmap Alignment

`docs/roadmap.md` §3 marks v0.2.x "✅ COMPLETE" with deliverables including "Improve error handling and logs ✅ (Unified error handling system implemented with type-safe error hierarchy, correlation IDs, and comprehensive test coverage)" and "Add basic documentation for architecture ✅". The unified error system is real and good (`errors/`). The architecture documentation claim is overstated — `core/README.md` documents an outdated four-component model and does not mention DI, sagas, unified caching, the notification subsystem, the audit logger, or diagnostics. `project-status.md` repeats "100% Functional" claims that don't match the workflow-engine and hook-system TODO stubs.

v0.4.x is the next milestone where this hurts: it promises "Comprehensive audit logs for actions" and "Comprehensive audit trail UI". The current audit layer is bolted onto API/CLI rather than to `RecordManager`, splits across two storage backends, and doesn't capture `user_id` in the DB path. The audit-trail UI cannot be cleanly built on this until core's record-manager itself writes coherent audit entries. v0.4.x also promises "Enhance hooks/events system for advanced module extensibility" — the hook system currently has a TODO for workflow-engine integration (`hooks/hook-system.ts:403`) and another for log writing. These are not v0.5 polish items; they are gaps in a milestone marked complete.

## Notes / Open questions

- The saga design is good and matches the manifesto's archival constraint (Git as authoritative, no rollback after commit). The concern is not the **what** of sagas but the **how-much** — building recovery, metrics, idempotency, resource locking, state persistence ahead of a single municipality pilot is a meaningful bet on operational scale that the project hasn't reached. Worth a human decision: keep as forward investment, or trim to executor + state store + idempotency only and defer the rest.
- `core/src/audit/` versus `database-service.logAuditEvent` versus per-saga hook emissions overlap in intent. A single human pass to define the audit-event contract (where it's written, what fields are required, who can read) would close finding **core-001** and **core-013** together.
- The DI container should be evaluated against actual testability needs. Several tests in `core/src/di/__tests__/` test the DI container itself rather than civic features; if those tests were removed alongside the container, would the rest of the suite still pass cleanly? If yes, the DI container is delivering test surface without test value.
- Many findings here (core-002, core-005, core-006, core-007, core-012) cluster around the same pattern: features were typed/exported/registered ahead of being implemented or wired. This is a recognisable Cursor-assisted code shape ("build the API surface first"). A follow-up cleanup pass that removes unused exports and converts TODO stubs into either real code or explicit `NotImplementedError` would tighten the surface considerably.
- The branch under audit (`broadcast-box`) has `modules/broadcast-box/` as the flagship, but core has zero references to it — no record types for recordings, no session-artefact contract. This means broadcast-box currently lives outside the manifesto's "Markdown as civic format" guarantee at the core level. Whether that's correct (broadcast-box should produce records that core ingests) or an oversight is a question for Phase 2 synthesis.
- Test counts disagree: `project-status.md` claims "Core: 160+ tests"; the actual count is 36 test files with ~8,435 LoC. Hard to map LoC to test counts without running them, but the order of magnitude is right. Worth verifying after the audit.
