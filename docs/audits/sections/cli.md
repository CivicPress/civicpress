## Technical Quality

**Code organisation.** The structure is clean at the macro level: one file per command in `cli/src/commands/`, registered from `cli/src/index.ts`, with shared utilities in `cli/src/utils/` (`cli-output.ts`, `global-options.ts`, `auth-utils.ts`, `record-locator.ts`). The pattern is consistent across nearly every command: read global options → initialise `cliOutput` → require auth via `AuthUtils.requireAuthWithCivic` → check permission with `userCan` → do work → emit `cliSuccess` / `cliError` → call audit log → `process.exit(N)`. This consistency is a real strength and makes the codebase easy to skim.

**Types and errors.** Output types are well-modelled (`CliSuccessOutput<T>`, `CliErrorOutput` in `cli-output.ts`). Error codes are reasonably consistent strings (`PERMISSION_DENIED`, `RECORD_NOT_FOUND`, `INVALID_STATUS`, etc.). However, action handlers are nearly all typed `(options: any)`, which defeats type-safety for the surface the user actually interacts with — the very layer most exposed to typo and option-mismatch bugs.

**Tests — the big problem.** `project-status.md` claims "120+ CLI tests passing at 95% coverage." The reality on this branch: `tests/utils/cli-test-utils.ts:145–156` exports a `runCivicCommand` that *returns a hardcoded `{stdout: '', stderr: 'CLI testing disabled in this environment', exitCode: 1}` without ever spawning the CLI.* The bulk of `tests/cli/init.test.ts`, `create.test.ts`, `list.test.ts`, `view.test.ts`, `edit.test.ts`, `commit.test.ts`, `status.test.ts`, `validate.test.ts`, `search.test.ts`, `diff.test.ts`, `export.test.ts`, `import.test.ts`, `history.test.ts`, `hook.test.ts`, `template.test.ts` then asserts `expect(result.stderr).toContain('CLI testing disabled')` — i.e., the tests *assert that the CLI is not exercised* and pass. That's 15 of 27 test files. There is real test coverage in `users.test.ts`, `storage.test.ts`, `geography.test.ts`, `audit.test.ts`, `config-init-status.test.ts`, `security-commands.test.ts` (these use a different pattern via `setupGlobalTestEnvironment` + `execSync`), but the headline number badly overstates what's exercised.

**Documentation.** `cli/README.md` lists 20+ commands; `docs/cli.md` is a partial guide. Both undercount the actual surface — the binary registers **55 commands** (counted via `grep -h "\.command(" cli/src/commands/*.ts`). Several commands have no README mention at all (e.g., `auto-index`, `cleanup`, `backup`, `notify:*`, `cache:*`, `diagnose:*` family, all 9 `storage:*` subcommands). `cli/README.md` also still claims "25+ endpoints" prose carried over from the API and references commands like `civic users list` that the code spells `users:list`.

## Security (LIGHT)

The CLI's main security concern is **auth surface**: every mutating command goes through `AuthUtils.requireAuthWithCivic` (which validates a session token against the local DB) and `userCan(user, '<perm>')`. That's a consistent gate — well done — and it short-circuits to `process.exit(1)` with a clear error code on auth failure. Production protections include the early-return guard on `auth:simulated` (`auth.ts:327` — `if (process.env.NODE_ENV === 'production') exit`), and the dual confirmation on `cleanup` (typing the org name + a y/n).

Concerns worth flagging for a dedicated security session:

1. **Tokens accepted via `--token <token>` flag.** Common pattern across the CLI. Tokens passed on the CLI end up in shell history and `ps`/`/proc/<pid>/cmdline`. A `--token-file` or `CIVIC_TOKEN` env-var-with-warning would be safer. (`auth.ts`, `users.ts`, `create.ts`, `list.ts`, `view.ts`, `commit.ts`, `edit.ts`, etc.)
2. **`auth:simulated`** is gated only by `NODE_ENV === 'production'`. A clerk laptop that runs without `NODE_ENV` set (the default) can issue admin tokens with `--username anyone --role admin`. Manifesto Trust would prefer a positive opt-in (`CIVIC_DEV_MODE=true`) plus a louder warning, since "production" is not the only sensitive context.
3. **`cleanup` command** wipes `data/`, `.civicrc`, `.system-data/civic.db`, `modules/api/data`, and `modules/api/.civic` from `process.cwd()` (resolved relative to `__dirname/../../../`). If a clerk ever runs `civic cleanup --force` from the wrong directory, the blast radius is the whole town. The confirmation is mitigation, not safety. (`cleanup.ts:38–51`).
4. **`create.ts:168`** sets `context.author = 'system'` with a `TODO`, then `recordData.author = user.username` is set correctly two blocks later. The dead TODO is harmless here but suggests the template-context author field is never threaded through — anyone reading template variables expecting `${author}` to be the actor will get `'system'`.
5. **Hardcoded record-type whitelist in `create.ts:77–86`** and **status whitelist in `status.ts:52–59`** — the project sells itself as "config-driven record types and statuses" (project-status.md says "Config-driven validation (RecordValidator loads from config.yml)"); the CLI silently overrides that with a fixed list. A clerk who configures a new type via `data/.civic/config.yml` will then have `civic create` reject it. This is also a Trust issue (claims vs. reality).
6. **Compiled `cli/dist/`** appears to be tracked in the repo according to filesystem state (dist files visible). Compiled artifacts in source control invite supply-chain confusion — a single misaligned commit ships stale code that does not match `src/`.

No SQL-injection-equivalent surfaces in the CLI itself (the DB layer is in core). No obvious secrets in source. Bcrypt rounds set to 12 in `init.ts` and `users.ts`, which is fine.

## AI-Generation Smells

This module has the signature look of Cursor-assisted churn — long, exhaustive but shallow.

- **Massive duplication in `init.ts`.** The 1,696-LoC `init.ts` defines the same `record_types_config` / `record_statuses_config` literal block at least three times (lines 76–165, 209–293, and within `setupCivicrcFromFile` / `setupCivicrcNonInteractive`). The org-config literal block is copy-pasted across three setup helpers (lines 920–963, 1010–1053, 1338–1382) with near-identical content. This is "the model rewrote the whole thing for each branch of the if/else" and is the single most refactor-needed file in the CLI.
- **Fake comprehensiveness in `diagnose.ts`.** `diagnose.ts` is 803 LoC. The bulk is hand-rolled `cliRaw` lines that branch on every possible `issue.details.*` shape (`missingTables`, `missingColumns`, `missingIndexes`, `extraTables`, `integrityResult`, `path`, `size`, `missing`, `missingDirectories`, `missingFiles`, `systemMemory`, `processMemory`, `loadAverage`, `note`…). Each rendering case is duplicated between `outputHumanReadable` (lines 417–653) and the verbose-mode check details path (lines 569–652). A renderer table or one `renderIssueDetails(issue)` function would collapse this to ~80 LoC.
- **Stub tests that document instead of test.** `tests/cli/init.test.ts` (and 14 others) contains a long parade of `it('should X (manual test)', ...)` cases whose entire body is: invoke the disabled mock, then assert the disabled-mock string. This is the textbook AI smell: lots of tests, plausibly named, that exercise nothing. The headline test count is therefore false confidence (Manifesto Trust).
- **`cleanup.ts` confirmation riddle.** The org-name confirmation is a creative bit, but the implementation (`packageJson.name.toLowerCase().replace(/[^a-z]/g, '')` → "civicpress") means the answer is "civicpress" for every project the user has not renamed. Looks safety-conscious; in practice it's a single trivial word. Smells like a model satisfying a "must require typed confirmation" requirement without thinking about realistic deployments.
- **`auto-index` demo command exposed in production CLI.** `auto-index.ts` is described as "Test and demonstrate auto-indexing workflow" with `--demo / --create / --update / --list` flags. The `--demo` runs a contrived create-modify-list scenario for the indexer. This is a developer test harness wired into the user-facing binary. Three `TODO: Fix type compatibility...` comments in `auto-index.ts:104, 199, 215` suggest it was never finished but never removed either.
- **`login` command with `--logout` and `--status` flags.** `login.ts` defines `civic login`, then handles `options.logout` and `options.status` inside the same action (lines 70–79). The user discovers logout by reading help text on `login`, which inverts every UX convention. Meanwhile separate `auth:login`, `auth:me`, `auth:password`, `auth:providers`, `auth:validate`, `auth:simulated` commands exist — six in the `auth:*` namespace plus the `login` toplevel — so there are two parallel auth UX trees and one of them is internally inconsistent.
- **`console.log` direct calls in `cache.ts`** (lines 60, 89, 192, 262) bypass the `cliOutput` abstraction the rest of the codebase uses. Any user running `civic cache:metrics --json` gets human-formatted output anyway. The rest of the codebase is disciplined; this one file isn't.
- **Redundant "explain what" comments.** Examples: `// Initialize CLI output with global options` (~30+ occurrences), `// Get configuration from central config`, `// Check view permissions`. These restate the next line. Harmless but they suggest the writer's reviewer was an LLM not a human.

## Architecture

**Boundaries are clean where it matters.** The CLI is a thin orchestrator: `cac` dispatches, `cli-output` formats, `AuthUtils` enforces, and all the real work is delegated to `@civicpress/core` (`CivicPress`, `AuthService`, `WorkflowConfigManager`, `TemplateEngine`, `RecordParser`, `RecordSchemaValidator`, `GitEngine`, `BackupService`, `IndexingService`, `AuditLogger`) and `@civicpress/storage`. That is the right shape: the CLI is a UI on top of core, not a parallel implementation.

**Internal cohesion is uneven.** `init.ts` blurs the boundary the most — it manually wires defaults, reads `core/src/defaults/`, calls `CivicPress.initialize()` twice, runs indexing twice (once normal, once with `syncDatabase`), all in one action handler. This is "saga-by-imperative-script," not the saga pattern. If any step fails midway, the half-initialised state isn't rolled back — only logged-and-continued. Manifesto Trust suggests this initial-state command, of all commands, should be transactional or at least idempotent on retry.

**Dynamic imports everywhere.** Most commands do `await import('@civicpress/core')` inside the action handler instead of importing at the top. The comment "Set logger options immediately to prevent warnings during config loading" in `cli/src/index.ts:39–45` hints why: top-of-file imports trigger config loading too early. This works but creates a startup-order coupling that is undocumented and easy to break.

**Configuration scope confusion.** `init.ts` writes to both `data/.civic/` (`config.yml`, `org-config.yml`, `roles.yml`, `workflows.yml`, `hooks.yml`, `analytics.yml`, `geography-presets.yml`, `templates/`) and `.system-data/` (`notifications.yml`, `storage.yml`, `civic.db`). The two-tier split is intentional ("system" vs "data" → vendor-portability), but `init` is the only place that fully understands both. The `cleanup` command knows only some paths and from a relative root (`process.cwd()`). The two commands aren't symmetric — `cleanup` cannot fully undo `init` (e.g., `.system-data/notifications.yml` is left intact, but `data/` is removed). For a clerk who hits "I want to start over," that asymmetry will be confusing.

**Backup and storage** are the bright spots: `backup.ts` cleanly delegates to `BackupService` with explicit options for `includeStorage`, `includeGitBundle`, `compress`, and `restoreStorage`. This is the kind of command a clerk could trust offline.

## Roadmap Alignment

`project-status.md` claims the CLI is "100% Functional," with "25+ commands," "120+ tests at 95% coverage," and lists `civic diagnose --fix`, sort options, unified error handling, etc., as complete deliverables. Reality check on this branch:

- **Command count is undersold**, not oversold: 55 commands are registered, not 25+. The status doc is stale.
- **Test count is dramatically oversold.** 15 of 27 CLI test files in `tests/cli/` are no-ops asserting the disabled-mock string. The 120+ test number includes these. Genuine end-to-end CLI coverage is concentrated in `users.test.ts`, `storage.test.ts`, `geography.test.ts`, `audit.test.ts`, `config-init-status.test.ts`, `security-commands.test.ts`, plus the small unit test at `cli/src/commands/__tests__/diagnose.test.ts` (which only verifies registration, not execution — see its comments at lines 144–148).
- **`civic diagnose --fix`** is real and well-wired (`diagnose.ts:701–778`), and is genuinely useful for v0.2.x "Add CLI improvements (diagnostics, repair, validation)" in `docs/roadmap.md`. PASS for that claim.
- **`docs/cli.md` lists options that don't exist** (e.g., `civic list --author`, `--format csv`, `--sort` — none of those flags are registered in `list.ts`). That's a Trust gap.

Roadmap milestone impact: the most important CLI roadmap commitment ahead is v0.5–0.8 "Make CivicPress usable by real municipalities" + "Write municipal documentation and onboarding guide." The findings below — especially Ease-of-Use ones — directly threaten that milestone. The CLI is not currently in a state where a clerk who has never used a terminal could follow `cli/README.md` and succeed.

## Notes / Open questions

- The `tests/utils/cli-test-utils.ts:runCivicCommand` stub deserves its own decision: either (a) wire up real spawn-based CLI tests, (b) delete the 15 placeholder files outright and adjust the test-count narrative in `project-status.md`, or (c) keep them as documentation but rename to `*.notes.md` and remove the `expect(...).toContain('CLI testing disabled')` assertion that creates fake passing tests. The current state — assertions that pass *because the command did nothing* — is the worst of the three.
- The pre-WIP version of CLAUDE.md (and project-status.md) says "CLI: 120+ tests, 95% coverage." After this audit a synthesis update should reconcile that line.
- The `info` command (line `info.ts:21`) has the right shape (org-config visible to public, system config gated to admins). It is one of the few commands that gets the public-good / public-by-default split right and could be cited as a template for the rest.
- Manifesto note: the CLI is the place where "if a clerk can't use it, it's not finished" is most concretely measurable. A small followup that runs `civic init && civic create bylaw "Test" && civic list && civic view bylaw/test && civic commit -a -m 'test'` as an actual happy-path smoke test, in CI, would catch most of the regressions the placeholder tests miss.
- Consider whether `civic init` should be split — `civic init --bootstrap` for the system bits (`.system-data`, `.civicrc`, admin user) and `civic init --records` for the data bits (`data/`, demo data, git init, indexing). The current monolith is doing too much in one transaction.
- The relationship between `civic login` and `civic auth:*` deserves a deprecation: pick one tree, alias the other, document the migration. The current split is the kind of thing audits exist to catch.
