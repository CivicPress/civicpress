## Technical Quality

Code quality cannot really be evaluated because there is no code. The module ships **one JSON Schema file** (`schemas/record-schema-extension.json`, 210 lines, well-formed draft-07), one README, and a stub `package.json` with no dependencies and no scripts. The schema itself is reasonable: properties are typed, descriptions are present, enums are bounded, the `allOf`/`if`/`then` pattern is used correctly to scope rules to legal record types, and nothing in it is obviously wrong.

The bigger issue is **mismatch between scaffolding and substance**. `docs/specs/legal-register.md` (a v1.0.0 "stable" spec dated 2025-07-03) promises a module with `LegalDocumentViewer.tsx`, `LegalDocumentEditor.tsx`, `LegalApprovalWorkflow.tsx`, `useLegalRegister.ts`, `legal-parser.ts`, `legal-validator.ts`, `legal-indexer.ts`, `core/legal-register.ts`, `core/legal-validation.ts`, `core/legal-workflow.ts`, `core/legal-indexing.ts`, a `legal-register/` test directory, CLI commands (`civic propose bylaw`, `civic approve …`, `civic archive …`), hooks (`onBylawProposed`, `onBylawAdopted`, `onBylawArchived`), GPG-signed legal documents, MFA, quorum verification, and a real workflow engine. None of this is implemented anywhere. The spec is essentially a wishlist that the implementation has not started. `docs/todo.md:354` confirms the work is still planned for v1.5.0.

**There are no module-internal tests.** The schema is exercised only indirectly via `tests/core/indexing-service.test.ts` and a few others where `'legal-register'` appears as a string in fixture metadata — those tests verify the indexing system can group records by module name, not that the schema actually validates anything. A targeted test that constructs a bylaw with and without `document_number` and asserts the extension's regex fires would take ~20 lines and is conspicuously absent.

A small-but-real **duplication concern**: every field declared in `legal-register/schemas/record-schema-extension.json` (document_number, legal_authority, jurisdiction, approval_date, effective_date, expiry_date, supersedes, superseded_by, retention_schedule, classification, public_access, approval_chain, change_history, etc.) is **also already declared in the base schema** at `core/src/schemas/record-base-schema.json` lines 342–605. The base schema's own README (`core/src/schemas/README.md` lines 89–104) explicitly calls these "Compliance Fields" including "Legal/Government metadata (document_number, legal_authority, jurisdiction, etc.)". So enabling the legal-register module does almost nothing the base schema doesn't already do; the regex on `document_number` (`^[A-Z]{2,4}-\\d{2,4}-\\d{1,6}$`) is the only additive constraint, and it's not enforced as required.

## Security (LIGHT)

There is no runtime code to attack. The schema itself is parsed by AJV in `core/src/records/record-schema-validator.ts`; that's where any vulnerability would live, not here. One small note: the schema-loader in `core/src/records/record-schema-builder.ts` lines 187–193 reads `process.cwd() + 'modules/' + moduleName + '/schemas/record-schema-extension.json'` directly. The `moduleName` value comes from `data/.civic/config.yml`'s `modules:` array. If a hostile config could inject path components (e.g., `'../../etc/passwd'`), this would attempt to read an arbitrary file as JSON; in practice the JSON parse would fail harmlessly, but the path is also not normalized or restricted to an allowlist. Worth flagging for the dedicated security session as a low-severity path-handling check.

The schema's `change_history` and `approval_chain` blocks describe audit-trail fields, but because nothing in the module enforces them, they cannot be relied on as a security/audit boundary. Anyone editing the Markdown by hand can omit, alter, or forge these arrays.

No PII, no secrets, no auth concerns — the module never sees any of those directly.

## AI-Generation Smells

The module is small enough that gross AI-smell patterns (over-abstraction, ten-class-deep hierarchies, fake comprehensiveness) cannot really land. That said, a couple of things look characteristic of Cursor-style scaffolding-without-finishing:

1. **The package.json is a stub.** `description: ""`, `author: ""`, `keywords: []`, `main: "index.js"` (no `index.js` file exists), and a default `"test": "echo \"Error: no test specified\" && exit 1"` script. This is a `pnpm init` default that was never edited. It hints the module was created and then abandoned, not actively maintained.
2. **A 300-line "stable v1.0.0" spec without code.** `docs/specs/legal-register.md` is dated 2025-07-03 with `status: stable`, lists named authors and reviewers ("Sophie Germain", "Ada Lovelace", "Irène Joliot-Curie" — all famous historical figures, almost certainly placeholders), and describes a sweeping module — but the only implementation is the schema file. The spec is plausibly AI-generated and was never reconciled with what was actually built. This is the most concerning AI-smell here because it pollutes the documentation surface with false claims about capability.
3. **Schema duplication of base-schema fields** without acknowledging it is consistent with an LLM happily restating a list it already saw, rather than a developer thinking "wait, the base schema already has this." A human author would likely have either (a) chosen to define these fields *only* in the module, or (b) put only the legal-specific *constraints* (e.g., the `document_number` regex, the jurisdiction `enum`) in the extension and left field declaration to the base.
4. The README claims "**Validates legal compliance fields**" but the schema makes none of them `required`. The `if`/`then` block at lines 176–208 says jurisdiction "is recommended" — a description string, not a constraint. The README overstates what the file does. This is a typical LLM "explain the goal, not the actual implementation" pattern.

## Architecture

The integration pattern is, on its own merits, **clean and worth keeping** — and arguably the most interesting part of the module:

- A module declared in `modules: [...]` in `data/.civic/config.yml` is picked up by `CentralConfigManager.getModules()` (`core/src/config/central-config.ts:402–424`).
- `RecordSchemaBuilder.mergeModuleExtensions()` (`core/src/records/record-schema-builder.ts:174–212`) iterates the list and, for each module, reads `modules/<name>/schemas/record-schema-extension.json` from disk and merges it into the base schema via JSON Schema `allOf`.
- A separate registry (`pluginSchemas` Map, lines 33–36) exists for runtime-registered plugin schemas via `RecordSchemaBuilder.registerPluginSchema()` (lines 274–283). This is a second, more flexible extension point, but legal-register does not use it.
- The merge cache is correctly invalidated on plugin (un)registration.

That's a reasonable schema-composition pattern that **does** demonstrate the manifesto §3.1 modularity promise: another developer could write `modules/permits/schemas/record-schema-extension.json` and have it picked up with minimal effort. But there are real seams to flag:

1. **Hardcoded `moduleName === 'legal-register'` logic** at `record-schema-builder.ts:219–236` inside `shouldApplyModuleSchema()`. This function decides which record types a given module schema applies to — but it does so by special-casing `legal-register` by name in core. Any new module would need a core-side edit to its `shouldApplyModuleSchema` function. That breaks the modularity promise: the module cannot declare its own scope; it must be allow-listed in core. The right fix is to put `appliesTo` metadata inside the module's schema file (e.g., a top-level `civicpress:appliesTo: ['bylaw', 'ordinance', ...]` field) and have the builder read it from there — the same way the runtime `pluginSchemas` registry already does (`appliesTo: (recordType) => boolean`).
2. **`process.cwd()`-based schema path**: `record-schema-builder.ts:187–193` constructs the module schema path from `process.cwd() + 'modules/' + name`. This means modules must live as folders in the monorepo's `modules/` directory at runtime — they cannot be real npm packages installed into `node_modules`, and the code will silently fail to find the schema if the API server is launched from a different working directory than the repo root. For a system that aspires to be "WordPress for governance," modules should be installable, not hard-pathed to a folder layout.
3. **No module manifest or capability metadata.** A `legal-register/package.json` is present but tells us nothing useful — no description, no metadata, no declaration of "this module contributes a schema extension, applies to types [...]". A real module pattern needs some manifest convention so core can discover modules without enumerating their internals.
4. **No initialization hook, no DI surface.** The module never registers anything programmatically — it's discovered only by its filename. Compare to the runtime `RecordSchemaBuilder.registerPluginSchema()` API, which is a more honest extension point. A future "real" module (one that ships hooks, CLI commands, UI components) would need a richer plugin lifecycle than this.

In short: the **schema-extension piece** is a real working extension point (small but functional), and a second developer *could* write a similar schema-only module by copying this one. But the broader "module" pattern that the manifesto promises — modules that ship UI + CLI + hooks + tests, like WordPress plugins or Odoo modules — is not demonstrated here. This module is, at best, a 5% slice of what the manifesto means by "modular."

## Roadmap Alignment

The mismatch between documentation and code is the main story:

- **`project-status.md`** lists "legal register module" under "Phase 3: Advanced Features" and "Long-term Vision → Civic Modules" — i.e., planned, not done. This is broadly accurate.
- **`docs/roadmap.md`** does not name the legal-register module at any milestone; the closest reference is "Civic-specific modules (voting, feedback, legal register)" in "Phase 3: Advanced Features" of `project-status.md`, which is not a roadmap document. The roadmap itself never mentions legal-register, voting, or feedback explicitly — they live only in `project-status.md` and `docs/todo.md`.
- **`docs/architecture.md:135`** lists `modules/legal-register/ # Legal document management` as if it were a substantive module. This is misleading — a reader of the architecture doc would think a working legal-document management module exists.
- **`docs/specs/legal-register.md`** is a `status: stable, version: 1.0.0` spec that describes an implementation that does not exist. A "stable" spec for unimplemented code corrodes Trust (manifesto principle).
- **`docs/todo.md:354`** correctly lists the Legal Register Module as a Medium Term task for v1.5.0 — this is the most honest doc.

Net: the schema-extension file delivers something real (~5% of the spec), the project-status and todo docs are honest about the rest being unbuilt, but the architecture doc and the spec misrepresent the state. None of this threatens an immediate roadmap milestone — but it does threaten the **credibility** of the architecture and specs docs, which the manifesto's Transparency principle directly depends on.

For the roadmap **forward**: if legal-register stays "schema only" forever, that's a defensible scoping choice — but the spec and architecture doc should be rewritten to reflect it. If a real module is intended for v1.5.0 as `docs/todo.md` claims, the schema-extension piece is a starter, not a complete deliverable.

## Notes / Open questions

- Real demo data does not use the `legal-register` module label. The actual bylaw record at `data/records/bylaw/2021/bylaw-286.md` carries `module: public-bylaws`, not `module: legal-register`, and uses none of the legal-register-specific fields (no `document_number`, no `legal_authority`, no `jurisdiction`). This suggests the module-aware indexing pathway is unexercised by the very dataset shipped with the project. Worth confirming whether `module: public-bylaws` is a stale relic or an intentional separate module.
- The spec at `docs/specs/legal-register.md` lists authors "Sophie Germain", "Ada Lovelace", "Irène Joliot-Curie" — famous mathematicians/scientists, almost certainly placeholders generated alongside the spec. Worth a documentation pass to remove obviously placeholder bylines from "stable" specs across the spec corpus, not just here.
- The architecture doc (`docs/architecture.md:935`) mentions "module-specific indexes (e.g., `legal-register/index.yml`)" — but this index file path is not the same as the schema-extension file path in the module. Worth confirming whether the indexing system actually emits a `legal-register/index.yml` somewhere on disk (a sample search did not find one).
- A clean way forward exists and is partly already implemented: the runtime `RecordSchemaBuilder.registerPluginSchema(name, schema, appliesTo)` API (lines 274–283) is the *right* pattern for this — modules declare their `appliesTo` predicate themselves. Migrating the file-based loader to use this same convention (via a per-module manifest) would resolve findings 002 and 005 in one move.
- Recommend: explicitly decide whether `legal-register` is a *deliverable module* (in which case build the workflow, the tests, the CLI, the UI panel) or a *schema-only convention* (in which case rewrite the spec and architecture entry to say so honestly, and call it `schema-extensions/legal/` rather than `modules/legal-register/`). Either decision is fine; the present middle-ground undermines both.
