## Technical Quality

The code is consistently styled (Black + Ruff with 100-col line length, `mypy` configured), uses `structlog` throughout for structured logging, and applies modern async-Python patterns. Connection-handling code in `services/connector/websocket_client.py` is **defensive to the point of being ornate** — single file of 1,432 lines with a connection lock, a health-monitor loop, a reconnect loop, a separate "send" failure-counter that promotes status messages to keepalive duty, four levels of try/except nesting, and re-enrollment-on-auth-failure logic. It is clearly evolved through repeated production-debug sessions rather than designed up front; comments like "Ping loop removed - status messages serve as keepalive" (line 235) document the archaeology.

The service catalog is broad: `services/capture/`, `services/encoder/`, `services/preview/`, `services/storage/`, `services/streaming/`, `services/connector/`, `services/device/`, `services/recording/`, `services/mixer/`, `services/health/`, `services/ap_mode/`. Each subdir has 1–8 files. That's appropriate for the scope, but several services are **tightly coupled through the constructor of `CommandHandler`** which takes 14 injected dependencies (`services/connector/command_handler.py:39-56`). The command-handler file itself is roughly 800+ lines and dispatches ~25 command actions via `elif` chain — a clear "router-grown-into-god-class" smell.

Test coverage: 30 test files; the project has a `.coverage` and `coverage.xml` artifact (~390 KB), and `pyproject.toml` requires `--cov=src` on every pytest run. No coverage % is published in the README. Tests use `pytest-asyncio` and `pytest-mock`; markers for hardware/RTMP separate "real-device" integration tests. The Makefile separates `test`, `test-unit`, `test-integration`, `test-fast`, `test-hardware`, `test-hardware-full`. Quality and discipline of testing is **better than the monorepo's broadcast-box module** — but the integration tests are mostly mocked.

Documentation under `docs/` is **voluminous** — 33 files. Many are aspirational (`docs/trust-and-evidence-model.md`, `docs/compliance-framework.md`, `docs/disaster-recovery.md`, `docs/observability-strategy.md`). The `docs/engineering-analysis.md` is meta: a self-assessment that closes with "All Gaps Addressed - Implementation specifications complete… 95% production-ready." This is the most striking AI-generation smell in the repo (see below).

## Security (LIGHT)

Reasonable starting posture: outbound-only WebSocket (WSS), bearer token in `Authorization` header AND as `?token=…` query (`websocket_client.py:209-212` — duplication suggests historical compatibility, not a hardening choice), JWT-style credentials persisted via `CredentialManager`. The `docs/security-considerations.md` is thoughtful but mostly prose, not implemented controls.

Specific concerns surfaced from a moderate read:

1. **Token in URL query parameter.** Tokens passed as `?token=…` will end up in webserver access logs, proxy logs, and possibly browser history if anything ever opens that URL. Standard anti-pattern. Combined with the `Authorization: Bearer` header, the URL one is redundant and risky.
2. **No certificate pinning.** TLS validation is default Python behaviour. For appliances deployed in municipal networks (where the local IT may MITM-inspect), this is probably fine, but for the resilient-archival promise it deserves a note.
3. **Enrollment code stored as plain config string in SQLite** (`services/device/enrollment.py:33-35`). The DB file under `data/state.db` has no documented encryption-at-rest. `docs/security-considerations.md` claims "Credentials stored encrypted in SQLite" — the code does not appear to encrypt them.
4. **Re-enrollment is automatic on `AUTH_FAILED`** (`command_handler.py:160-184` and `_attempt_re_enrollment`). The device re-POSTs its enrollment code and device UUID to recover. If the enrollment code is long-lived (the audit didn't reach the server-side rotation logic), a compromised box can re-enroll silently. Worth pairing with the server-side audit (target 6).
5. **AP-mode web server** runs on a configurable port (default 8443) with a Nuxt-built static UI bundled into `src/broadcast_box/web/static/`. The `service.py` for AP-mode has a `timeout_minutes = 0  # Disabled for testing` — auto-deactivation is **disabled by default** in production code. An AP-mode window left open is a wifi attack surface.

Surfaces that need a dedicated security session: (a) enrollment-code lifecycle (both sides), (b) cred storage at rest, (c) the upload protocol (not seen this audit), (d) AP-mode web UI authn (not seen this audit), (e) firmware/OS update path (not designed).

## AI-Generation Smells

The repo is the most obvious case of AI-assisted writing among the audit targets. Surface signals:

- **README and docs use the emoji-section convention** (`✅`, `⚠️`, `🚧`, `🎯`, `🌟`) heavily — `agent/goals.md` is built almost entirely around emoji headers (`## ✅ Completed Foundation`, `## 🎯 Next Phase Goals`, etc.). Combined with sub-section formatting it reads as Cursor/GPT output.
- **`docs/engineering-analysis.md`** is a self-graded "senior engineer review" that opens with **"Analyst Perspective: Top 0.1% Senior Engineer"** and closes with **"95% production-ready. All Gaps Addressed."** This is performative self-grading, not a real audit — it is the kind of output an AI generates when asked to "review like a senior engineer." It is also the centrepiece of the project's own credibility claim.
- **33 documentation files for a 22k-LoC alpha project**, many naming themselves like enterprise artifacts (`compliance-framework.md`, `disaster-recovery.md`, `capacity-planning.md`, `accountability-roles.md`, `trust-and-evidence-model.md`, `opinionated-vs-configurable.md`, ADR folder, archive folder). The breadth of these documents is wildly out of proportion to the maturity of the code (single `[Unreleased]` entry in CHANGELOG, no `[0.1.0]` shipped). This is **fake comprehensiveness**: documentation aspiring to credibility, faster than implementation can earn it.
- **`websocket_client.py`** at 1,432 lines is a textbook AI-debugged file — every error path has been instrumented with `logger.error(... exc_info=True)` and multiple try/except layers added one bug at a time. The code _works_ but has 3–4 redundant disconnect-recovery paths (`_handle_disconnect`, `disconnect`, the health monitor in `_health_monitor_loop`, the per-message `except` blocks). A human-architected version would unify these.
- **`command_handler.py`** routes 25 commands via `elif` chain (lines 190–234). The constructor injects 14 services. This is what happens when an LLM is asked to "add a command for X" repeatedly without a refactor pass.
- **Speculative knobs**: `pyproject.toml`'s `tool.mypy` has `disallow_untyped_defs = false` (off!) and `check_untyped_defs = true` — i.e. typing is half-enforced. Many functions have type hints; many don't. Classic AI-edit-by-edit accretion.
- **Duplicated WebSocket-message format-handling**: `websocket_client.py:486-565` has _three_ branches handling commands — `original_type == "command"` (standard), `original_type == "control"` (legacy), `original_type is None` (server format without type field). The hardware is bending over backwards to accept whatever the server sends, including malformed forms. This either reflects a confused contract or AI accreting compatibility for cases that never happened.

The **agent/** folder is itself the most telling artifact: `agent/README.md` declares this is a "common base memory for any AI agent (Cursor, Copilot, etc.) working on the CivicPress Broadcast Box project" — i.e. the repo is structured to be agent-driven by design. That is not bad on its own, but it explains the doc/code shape.

## Architecture

The runtime topology is sound: a single `BroadcastBoxApp` (`src/broadcast_box/main.py`) wires ~20 services in a constructor-injection style — no DI container, just hand-wiring. The event bus (`event_bus.py`) is a typed pub/sub with priorities, used to keep services loosely coupled. Capture is unified via FFmpeg with "encode once, fan out many" to file/RTMP/WebRTC outputs — that's the right design (cited in the CHANGELOG as a recent refactor away from 8 specialised restart methods to one `_update_capture_outputs`).

The split into `services/connector/` (WebSocket protocol, command handler, status reporter), `services/capture/`, `services/encoder/`, `services/preview/`, `services/storage/`, `services/streaming/`, `services/recording/` reflects clear boundary thinking. The lift from "OpenCV-based capture" to "FFmpeg unified capture" (per CHANGELOG) is a real architectural improvement.

That said, the contract layer is doing too much. `CommandHandler` is a 14-dependency router. The protocol-message-shape juggling in `WebSocketClient._receive_loop` shouldn't be in transport — it should be in a thin protocol-adapter layer. The repository missing-a-protocol-adapter explains why both sides of the system have grown duplicated, defensive shape-coercion code.

There is no `core/`-style shared library between the hardware and the software sides — each defines its own message types. Both sides have "ErrorCode" enums. Both define "Command", "Event", "Ack" type names. A shared protocol-types package (Python + TypeScript, or just the JSON schema) would have prevented this audit's #1 finding.

## Roadmap Alignment

- **broadcast-box does not appear on `docs/roadmap.md`** in the monorepo, and the hardware repo's `agent/goals.md` is its own ad-hoc roadmap that **bears no relation** to the v0.2.x → v0.3.x → … → v1.0 phases in the main `docs/roadmap.md`. This means the flagship's hardware side is **invisible to the project's published roadmap**, which is a Transparency-principle problem.
- The hardware's CHANGELOG declares `[0.1.0] - TBD` (i.e. nothing shipped yet), while the README says "Core implementation complete." These two claims need reconciling — they suggest the team is operating from internal momentum without a public version-release discipline.
- `project-status.md` in the monorepo does not mention the hardware repo at all (search the document). Project-status as a whole is dated "January 2025" and tracks v0.2.x deliverables that are about records management — broadcast-box belongs in a section that doesn't exist.

Net: the flagship is not on the roadmap, the hardware side is not on the project-status, and both sides have their own internal "complete!" claims. This is the **single biggest manifesto-Transparency gap** in the project as a whole.

## Notes / Open questions

- **The contract question, summarised.** The hardware-side protocol document and the implementation disagree on a non-trivial point (top-level vs nested `action`). The implementation accepts three different message shapes defensively. The server (target 6) sends one specific shape. The hardware's protocol doc describes a fourth, idealised shape. There is no single canonical contract artifact. **A shared protocol-spec file (JSON schema or `.proto`) consumed by both sides is the recommended fix.**
- **The civic-artifact gap is the deepest manifesto issue.** Once the hardware-side recording-end pipeline produces only an `.mp4` (plus _intended_ JSON sidecar that isn't even consistently written on this machine), the CivicPress server has to invent the civic-record layer on top. That work isn't in the monorepo either. The flagship's central manifesto promise — "public meetings remain public records" — has no implemented path from media blob to Markdown civic record.
- **The "95% production-ready" doc.** Recommend deletion or full honest rewrite in a follow-up. It is the most quotable misleading artifact in the codebase and undermines every other doc.
- **No license = no pilot.** Before any municipality runs this, the license needs to be set (Apache-2.0, AGPL-3.0, or MPL-2.0 are the obvious candidates for civic infrastructure).
- **Pi support is ambiguous.** Deployment guide says x86_64 (Intel N100 / i3); `Makefile` has `PI_HOST` deploy targets. Either Pi is supported and deployment-guide is wrong, or Pi is dev-only and Makefile is misleading. Worth clarifying for equity reasons.
- **Upload protocol unseen this audit.** The HTTP upload destination (chunked upload, manifest format, resume-from-byte-offset semantics) is the other side of the resilient-archival promise and was out of scope at moderate depth. Recommend including it in the dedicated hardware deep-dive next session.
- **A unified contract package** (Python + TS shared types, or a JSON Schema package consumed by both repos) would address findings BB-HW-001 and BB-HW-004 simultaneously and is the single highest-leverage architectural fix.
