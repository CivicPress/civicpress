## Technical Quality

The site is small — 4 source files, 1,414 LoC total. Code is readable, Vue 3 Composition API with `<script setup lang="ts">`, properly typed with the local `tsconfig.json` referencing the four `.nuxt/tsconfig.*` shards. There are no inline scripts beyond the Plausible bootstrap (author-controlled), no API calls, no auth state, no client-side data fetching — it's essentially a content site with locale switching. That radically lowers the risk surface compared to the monorepo's `modules/ui/` (29,300 LoC, 112 files).

**That said, `app/pages/index.vue` is a single 1,020-line god-page** — the same anti-pattern flagged in ui-008 for `RecordForm.vue` (1,310 lines). The page interleaves 12 sections (hero, demo-preview, why-intro, why-video, municipal-leaders, features, how-it-works, values, roadmap, governance, founder-note, FAQ, CTA) with two full inline SVG blueprint backgrounds (light + dark, ~150 lines each) and three additional inline SVG decorations for values/roadmap sections. Decomposing into `<LandingHero>`, `<LandingFeatures>`, `<LandingRoadmap>`, etc. would (a) reduce per-edit cognitive load, (b) make the SVG backgrounds an extracted `<BlueprintBackground>` component reused 5x rather than copy-pasted, (c) bring the site in line with its own `agent/conventions.md` "Maintain a single landing route (/) initially; plan sections via layout components (LandingHero, etc.)" — which the page demonstrably violates.

**Test coverage is zero.** `agent/memory/decisions.md` records the explicit decision "2025-11-10: Defer automated tests initially — Single-page marketing site with limited interactivity." That's a defensible call for a static landing page, but the README's "Tech Stack" mentions no testing posture, the agent's `current-session.md` lists no test-related blockers, and there's nothing that would catch a regression on the i18n key extraction logic (`heroBadges` and `municipalLeadersCards` in `pages/index.vue:4-41` both manually unwrap Nuxt i18n's message-AST objects via `item?.body?.static ?? ''`, which is exactly the kind of internal API contract that breaks across i18n version bumps with no visible warning).

**Type safety has the same `: any` pattern the monorepo UI uses.** `pages/index.vue:14,29` annotate `node`/`card` as `any` with `eslint-disable-next-line @typescript-eslint/no-explicit-any` to extract from i18n message AST. That's a small surface, but the comment signals the project's standard escape hatch is "disable the lint and cast." If the AST shape changes in `@nuxtjs/i18n` 11.x, the badges and municipal-leaders cards silently render as empty strings — no test would catch it.

**Three deps in `package.json` are imported nowhere in source**: `marked 16.1.1`, `leaflet 1.9.4`, `@vueuse/core 13.5.0` (the last is transitively used by `@nuxt/ui` but listed redundantly). `@nuxt/scripts ^0.13.0` is installed but Plausible is loaded via raw `useHead({ script: [...] })` rather than `useScript()` — meaning a real consent-aware integration would require both rewriting and shedding the dep, OR adopting `useScript()` and dropping the manual code. `@nuxt/eslint`, `@nuxt/test-utils`, `vue-tsc` are listed as **runtime dependencies** rather than `devDependencies` (`package.json:14-16,18,27`) — they'll be installed in production deployments unnecessarily. Bundle-impact is zero (static site), install-time impact is small but real.

## Security (LIGHT)

**No XSS surface.** Unlike the monorepo UI, this site has no `v-html` rendering of user data, no markdown pipeline, no API-fetched HTML injection. The lone `innerHTML` (`app/app.vue:69`) is the inline Plausible init script — author-controlled, no user input flows into it. The site cannot be turned into a token-theft vector because there are no tokens to steal: no auth, no `localStorage` JWT, no CSRF token.

**Plausible Analytics script source.** Plausible is loaded from `plausible.io` directly (`app/app.vue:66`). If `plausible.io` is compromised or DNS-hijacked, every visitor to `civicpress.io` would execute attacker code. The mitigation `@nuxt/scripts` (already installed!) offers — `useScript()` with SRI hashing, async loading guarantees, and trusted-source verification — is unused. For a "transparency by default" civic site, adding a Subresource Integrity hash to the Plausible script tag is a 1-line fix.

**Missing security headers.** `nuxt.config.ts` does not set `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, `Referrer-Policy`, or any of the modern security headers. GitHub Pages does NOT set these by default. Anyone running a Lighthouse / Mozilla Observatory / SecurityHeaders.com scan against `civicpress.io` will see a low/F-grade score for headers — exactly the trust-eroding fingerprint a journalist or municipal IT contact would pick up first. The hardest constraint here is that GitHub Pages can't set custom headers; a Cloudflare proxy or a Netlify/Vercel host could. This is more a hosting-decision finding than a code-fix.

**No CNAME committed.** The agent decision-log `2025-11-10` says "Target GitHub Pages via Nitro `github-pages` preset… need for `public/CNAME`" — but `find site/public -name CNAME` returns nothing. The deployment will lose the `civicpress.io` custom domain on every CI run unless the CNAME is configured externally in the GitHub UI (which means it's not reproducible from the repo). For an open-source project that wants its deployment to be inspectable and rebuildable, the CNAME absence is a hidden infrastructure dep.

**No CI/CD workflow committed.** `find site -name "*.yml" -not -path "*node_modules*" -not -path "*.nuxt*"` returns only `pnpm-lock.yaml`. The README says "Build the site: `pnpm generate`. Deploy the `.output/public` directory to GitHub Pages." — manually. There is no `.github/workflows/deploy.yml`, no automated build, no commit-to-deploy pipeline. A maintainer change of hands means whoever runs `pnpm generate` locally and pushes the dist branch becomes the only person who can publish. This is operational fragility, not a code bug, but it threatens the manifesto's "publicly inspectable" promise — the deployed site is not built from any commit on `main` in a verifiable way.

## AI-Generation Smells

**README claims "Nuxt UI Pro" but the project uses free `@nuxt/ui`.** README line 19 ("Framework: Nuxt 4 with Nuxt UI Pro"), line 30 ("Modern UI: Built with Nuxt UI Pro components"), `.cursor/rules.md:6` ("Deliver a static Nuxt 4 + Nuxt UI Pro landing page"), `agent/conventions.md:4` ("Nuxt UI Pro Landing template"), `agent/memory/project-state.md:12` ("Nuxt 4 + Nuxt UI Pro scaffold"), `agent/memory/decisions.md` row dated 2025-11-10 ("Use Nuxt 4 + Nuxt UI Pro Landing template"). Five different files documenting a paid dep that isn't installed. Most likely path: agent scaffolded the project with the Nuxt UI Pro starter, the actual code drifted to free `@nuxt/ui`, and *none of the prose docs were updated*. Cursor's "remember the original conventions, don't notice they changed" failure mode.

**`pages/index.vue:43` ships a hardcoded `const showDemoSection = false`** with a full 25-line `v-if="showDemoSection"` demo-preview section beneath it that includes a "Demo video coming soon" placeholder. The section is dead at runtime, present in the source, and the corresponding i18n keys (`landing.demoPreview.title`, `description`, `videoAlt`) are populated in both EN and FR for content that never renders. Either ship the demo or remove the section + translations.

**Mixed `process.client` and `import.meta.client` in the same file.** `AppHeader.vue:13,23` use the Nuxt 3 idiom `process.client`; lines 36, 42, 49 use the Nuxt 4-preferred `import.meta.client`. Five guards, two conventions, one file. The same smell flagged as ui-016 in the monorepo (33 `process.client` calls). Likely two AI editing passes that never reconciled.

**Three orphan deps in `package.json`** — `marked 16.1.1`, `leaflet 1.9.4`, `@vueuse/core 13.5.0` — none of which is imported anywhere in `app/`. `marked` is the *exact* dep that powers the monorepo's `useMarkdown.ts` XSS vector (ui-001). The site doesn't render Markdown at all. Leaflet is a 150KB map library; the site shows no maps. These look like dependencies copy-pasted from the monorepo's UI or from a scaffolding template that the agent never trimmed. Same "fake comprehensiveness" cluster the master audit flagged for storage, broadcast-box, core, notifications.

**i18n message AST unwrap pattern.** `pages/index.vue:9-21` and `pages/index.vue:23-41` each manually walk Nuxt i18n's internal message-AST shape (`node?.body?.static ?? ''`) with eslint-disable comments. This is the classic "ask Cursor for a hero badges array; it produces a workaround for a library internal it doesn't fully understand" smell. The correct API is `tm()` returning the array directly, or `t()` with `@@PluralRule` syntax. The workaround works today but will silently break on i18n module updates.

**Five inline SVG blueprint backgrounds in `pages/index.vue`** (lines 67-206 light, 137-206 dark for hero; ~100 lines each for values, roadmap). They're 90%+ identical between light/dark pairs. No extraction into a component. This is "AI generates the whole template, including decoration, and never refactors decoration into reusable parts" — the same pattern as the 4 EmailChannel reimplementations in notifications-005.

**`appVersion: '0.1.2'` in `nuxt.config.ts:13`** while landing copy advertises "Current Milestone – v0.2.0" in both EN and FR (`landing.roadmap.currentTitle/currentBody`, `landing.faq.items.0.answer`). The version number in the runtime config disagrees with the version number on the page. The monorepo UI has the same problem (ui-017: `appVersion: '0.1.3'` while the repo is v0.2.0). Two unrelated repos with the same AI-touched drift.

## Architecture

The architecture is appropriate to the scope: static-generated Nuxt 4 single-page landing site, no backend, no auth, no API consumption. `app.vue` does the head management + footer chrome, `AppHeader.vue` does the sticky nav + locale switcher + theme switcher, `pages/index.vue` is the entire landing page. `i18n.config.ts` is a 7-line plural-rule shim. `nuxt.config.ts` is 41 lines. There's no `composables/`, no `components/` beyond the two listed, no `layouts/` (uses default), no `stores/`, no `plugins/`. For a marketing page, this is correct minimalism.

**Integration with the rest of the ecosystem is via external GitHub links only.** The site does NOT consume the monorepo's API, does NOT share code with `modules/ui/`, does NOT import from `core/`. Footer links go to `github.com/CivicPress/civicpress`, `github.com/CivicPress/manifesto`, `tally.so` form, `mastodon.social`, `mailto:hello@civicpress.io`. The hero/CTA "Try the demo" goes to `demo-en.civicpress.io` / `demo-fr.civicpress.io` (which would presumably be a deployment of `modules/ui/`). This is the *right* architectural boundary: marketing copy lives in the marketing repo; product UI lives in the product repo; no code duplication.

**Relationship to monorepo UI: zero code overlap.** Searched: no shared components, no shared composables, no shared types. The monorepo UI's footer/header chrome (UDashboardNavbar, UDashboardSidebar) is the wrong abstraction for a public marketing site; the site correctly uses the simpler UHeader/UMain pattern. No content duplication either — the monorepo's `modules/ui/app/pages/index.vue` (a 1,007-key i18n citizen landing) and the site's `app/pages/index.vue` serve different audiences (logged-in clerks/citizens vs. cold-arriving evaluators). One small concern: there is no link from this site to the monorepo's running UI other than via demo subdomain. A municipal IT person wanting to read the source before adopting must navigate to GitHub manually.

**Hosting / deployment posture.** Target = GitHub Pages (per README, agent decisions, `nitro.preset = github-pages` is the implied target). GitHub Pages is a free, public, mostly-static-CDN service — manifesto-aligned for an open-source civic project. **However:** GitHub Pages is owned by Microsoft. A strict reading of "no vendor lock-in" might prefer Codeberg Pages, Cloudflare Pages, or self-hosted Caddy. Pragmatically, GitHub Pages is fine. The bigger issue is that the deployment is **not reproducible from the repo** (no CI workflow, no CNAME). Anyone forking this repo and trying to deploy a French-Canadian municipality variant at `civicpress.qc.ca` would need to set up the entire GitHub Pages + custom-domain dance from scratch with no documentation.

## Roadmap Alignment

The site is the **public mirror of the monorepo's roadmap and project-status documents** — which the master audit identified as stale. The site faithfully reproduces those staleness issues to a public audience:

- **"v0.2.0 marks the first stable foundation"** (FAQ items.0, roadmap.currentBody) — the master audit found "v0.2.0 Alpha / Production-Ready" claim in `docs/project-status.md` is materially overclaimed (17 Criticals open).
- **Broadcast-box is invisible.** The flagship per the user is broadcast-box (council-meeting livestream + recording). The site's `landing.features.items[1]` is titled "Every meeting, visible. ... From agendas to minutes and recordings, keep council sessions in one place" — that's the broadcast-box value proposition, but the word "broadcast-box" appears zero times. A municipal evaluator landing here cannot tell that a hardware device exists, cannot tell that there's a separate `civicpress-broadcast-box` repo, cannot tell that this is the project's lead feature. The site treats broadcast-box exactly the way the manifesto and roadmap do: implicit, unnamed, present only as a vague capability promise.
- **Ledger is also invisible.** Manifesto §3.5 names "Ledger as Civic Accountability" as a flagship. The site does not mention Ledger anywhere. So the public site silently agrees with what the master audit found: Ledger is no longer the flagship in practice. But neither does it announce that, leaving the manifesto reader confused.
- **Legal-register is named explicitly** as `tag: "legal-register"` on `landing.features.items[2]` "Every law, traceable." The master audit found legal-register has 7 findings and **zero in-module tests** (legal-register-004); the spec claims `status: stable, version: 1.0.0` for a module that barely exists. The site advertises legal-register as a working civic capability.
- **"Pilot projects with municipalities in Québec exploring pilot implementations"** (`landing.roadmap.pilotsBody`) — promising language with no public list of which municipalities, no case study, no contact for a journalist or another mayor to verify. If pilots are real, naming them (with consent) would deepen trust; if they're aspirational, the language should be softer.

**The site is the canary for the master audit's "documentation that doesn't reflect intent erodes trust" finding** (§ "Meta-finding"): the manifesto, the roadmap, project-status, AND the public website all simultaneously reproduce the same stale picture. A clerk who reads the manifesto, then the site, then runs the demo, walks through three documents that disagree subtly with the code. The site is the *easiest* of the three to update — but it has not been.
