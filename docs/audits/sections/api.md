## Technical Quality

The API has the bones of a competent Express service — a single `CivicPressAPI` class with explicit lifecycle (`initialize`, `start`, `shutdown`), centralised request-id and correlation-id middleware (`middleware/error-handler.ts:34-56, 132-220`), a `categorizeError()` helper that maps civic-domain error codes to HTTP statuses, and a `sendSuccess`/`handleApiError` utility pair that *most* routes use (`utils/api-logger.ts`, called in 101 places). The CSRF middleware is reasonable: it skips safe methods, Bearer-token requests, and explicit bypass headers, and validates an HMAC-style token from core (`middleware/csrf.ts`).

But the surface is uneven. Two of the largest route files — `routes/records.ts` (1,459 LoC) and `routes/users.ts` (1,443 LoC) — are nearly monolithic and contain heavy per-handler duplication (every records endpoint repeats the same `if (!user) throw 401`, the same `audit.log({...})` success/failure pair, the same `(req as any)` casts). Type discipline is weak: `grep "as any"` returns **503 occurrences** in `modules/api/src/`, including in hot paths like `routes/records.ts:606-650` where `user.username` is type-laundered through nested `any` extraction loops. The `AuthenticatedRequest` type (`middleware/auth.ts:6`) is the entire reason `as any` is supposed to be unnecessary — but routes ignore it and cast anyway.

Three route files are stubs that pretend to work: `routes/hooks.ts` (108 LoC), `routes/workflows.ts` (108 LoC), and `routes/export.ts` / `routes/import.ts` (24 LoC each). All four respond `200 OK` with hard-coded fake payloads (e.g. `workflows.ts:36-46` returns `{ id, name: 'Sample Workflow', status: 'active' }` for any id; `import.ts:14-22` returns `{ message: 'Import functionality', formats: [...] }` and never imports anything). These are mounted into the real router tree, are documented in OpenAPI, and pass their auth gate — so they *look* live to a caller. The README and `project-status.md` both claim "25+ endpoints"; the literal endpoint count is inflated by these stubs.

There is also a second, dead authentication middleware: `middleware/jwt-auth.ts` (228 LoC) defines `jwtAuth`, `requireRole`, `requirePermission`, and `optionalAuth` — none of which are referenced anywhere except their own file (confirmed via grep). It looks like an abandoned earlier auth design that was forked into `middleware/auth.ts` and never deleted. The duplicate `requirePermission` and `optionalAuth` exports are a footgun for anyone refactoring. Test coverage of the API surface comes mostly from `tests/api/*.test.ts` (18 supertest-driven integration files) plus 3 module-internal tests (only `error-handler`, `api-logger`, and `diagnose` have unit tests inside the module); there are no per-route handler unit tests for any of the 26 route files.

## Security (LIGHT)

The prior broadcast-box audit (`docs/broadcast-box/CLEANUP-AND-TEST-AUDIT-REPORT.md`, Jan 2025) flagged "50+ TODO comments, mostly auth/permission related." The current state across the API module is much better than that snapshot suggests: `grep -rn "TODO\|FIXME\|XXX\|HACK"` in `modules/api/src/` returns **5 hits total**, only two of which are auth-related (`routes/auth.ts:233` — "Implement proper session invalidation if needed"; `routes/records.ts:719` — a deprecation note about a legacy `PUT /:id` endpoint). The auth middleware is wired to most write-side mutating routes (records `POST/PUT/DELETE`, users mutations, storage uploads, indexing, history, audit, notifications, config writes), and authorization uses core's `userCan(...)` system. **Sampled 5 random endpoints**: `GET /records` (optionalAuth, intentional), `POST /records` (authMiddleware + requireRecordPermission('create')), `DELETE /records/:id` (authMiddleware + requireRecordPermission('delete')), `GET /audit` (authMiddleware + requirePermission('system:admin')), `POST /storage/files` (optionalAuth at mount + requireStoragePermission('upload') at route) — all four mutating ones are gated. The records router also emits `AuditLogger` events on both success and failure for create/update/delete/publish/status-change — that is genuine Transparency wiring.

That said, three concrete issues warrant a dedicated security session: **(1)** `routes/info.ts` constructs a fresh `CivicPress` instance per request, calls `civic.initialize()`, validates the token, then `civic.shutdown()` (lines 29-45). This is a denial-of-service amplifier — any unauthenticated request to `GET /api/v1/info` spins up the full core (DB connections, indexing service, Git engine) and tears it down. A trivial loop saturates the server. The injected per-request `civicPress` is already available on `req`, so this whole block is a structural mistake. **(2)** `routes/validation.ts` uses `requirePermission('records:view')` on every endpoint but is mounted in `src/index.ts:303` with **no upstream `authMiddleware` or `optionalAuth`** — so `req.user` is never populated and every request returns 401 regardless of token. Either the endpoint silently 401s in production (functional regression), or tests rely on a test-setup that injects users out-of-band (false coverage). Both are bad. **(3)** Similarly, `routes/status.ts` reads `(req as any).civicPress` at lines 27, 110, 195 and throws `"CivicPress not initialized"` if absent, but the mount at `src/index.ts:281` (`apiPath('status'), createStatusRouter()`) injects neither `civicPress` nor auth. Status endpoints will throw 500 at runtime under non-test conditions.

The CSRF middleware has one subtle skip: `middleware/csrf.ts:42-44` bypasses CSRF whenever the request carries `X-Mock-User`, with a "safe because X-Mock-User is only used in tests" comment. That's only true if `BYPASS_AUTH=true` is never set in production — and there is no runtime guard preventing a misconfigured deployment from accepting `X-Mock-User` and bypassing both auth (`middleware/auth.ts:19`) and CSRF. The `simulated` auth endpoint is correctly disabled when `NODE_ENV === 'production'` (`routes/auth.ts:248-256`), so the bypass surface is at least environment-gated, but a defence-in-depth check that asserts `NODE_ENV !== 'production'` in the CSRF/Auth `X-Mock-User` paths would be cheap and correct. Finally, `express-rate-limit`, `helmet`, and `compression` are listed in `package.json` but **never imported anywhere in `src/`** — the README at lines 605-607 advertises rate limiting ("100 requests per 15 minutes per IP, configurable via options") that does not exist in the running code. A clerk reading the docs would believe the API is rate-limited; it is not.

## AI-Generation Smells

Strong signal of mechanically generated code. **(a)** The four stub routers (`workflows.ts`, `hooks.ts`, `export.ts`, `import.ts`) follow a copy-pasted shape that looks generated together: each has identical imports, identical `requirePermission(...)` gates, identical hand-back of `{ message: 'X functionality' }` payloads, and identical OpenAPI-style JSDoc — but no real logic. They register as legitimate endpoints, which is exactly the "fake comprehensiveness" smell called out in the audit brief. **(b)** `middleware/jwt-auth.ts` is a dead 228-line duplicate of `middleware/auth.ts`; same function names, same shapes, never imported. It reads like an AI alternative draft that was never deleted. **(c)** The 503 `as any` casts and the type-laundering loop in `routes/records.ts:627-650` (a 23-line nested cast that tries to extract `username` from a possibly-object-shaped `user.username` field, falling through three `typeof` checks and a key-iteration loop) is exactly the shape of LLM-generated defensive code: the right answer is "fix the type at the boundary," not "guard against every shape the LLM has seen in training data."

**(d)** `routes/templates.ts:36-41` constructs a **new `TemplateService` per request** with `enableWatching: true`, meaning every call to `GET /templates` allocates a fresh file watcher; over time these leak file descriptors. The `// Get TemplateService instance from request` comment is a textbook "comment that explains WHAT instead of WHY". **(e)** `routes/info.ts:29-45` is even worse — it instantiates the entire `CivicPress` core per request just to validate a token, when the request already has a fully-initialised `civicPress` on it (every other route uses `(req as any).civicPress`). This is the kind of mistake an LLM makes when it doesn't have the file-level context to know there is already a shared instance. **(f)** Three out of four "minor" routers (`workflows`, `hooks`, `export`/`import`) reuse the exact same response shape and error-handling pattern, which is benign on its own but combined with the empty payloads strongly suggests `Generate me a workflows router that looks like the hooks router` rather than principled design.

**(g)** Redundant error-message string-matching in `routes/uuid-storage.ts:307-321`: the code inspects `errorMessage.toLowerCase().includes('size') || ... .includes('limit') || ... .includes('exceeds')` to *guess* whether an internal error means "payload too large" — instead of typing the storage error with a code. This is the antipattern of substring-sniffing error messages. **(h)** Heavy duplication of validation-error handlers — `routes/records.ts:21-43` defines `handleRecordsValidationError`, `routes/geography.ts:54-65` defines its own `handleValidationError`, `routes/uuid-storage.ts` uses `handleStorageValidationError`, and `utils/api-logger.ts` already exports a generic `handleValidationError`. Five different shapes for the same job, each invented per route file, each subtly different in field names ("details" vs "errors" vs "message"). **(i)** OpenAPI doc claims v0.1.3 (`routes/docs.ts:12`) and lists a `status: 'active'` enum (`docs.ts:48`) that does not appear anywhere in the actual record-status configs — clear evidence the OpenAPI spec was generated at one point and never reconciled with the live config-driven status system documented in `project-status.md`.

## Architecture

Boundary discipline is mostly correct but inconsistently enforced. The API talks to core through three layers: (1) direct imports from `@civicpress/core` (managers, config, auth service, audit logger), (2) the `CivicPress` orchestrator injected per-request via the `civicPress` property, and (3) a thin `RecordsService` wrapper (`services/records-service.ts`, 1,760 LoC) that adapts the API contract to the core record-manager API. Saga usage is delegated to core: `services/records-service.ts:1691-1699` invokes `recordManager.publishDraft(...)` which internally constructs a `sagaExecutor`. The API does not orchestrate sagas itself — it just calls core methods that happen to use sagas. This is the right boundary, but it means the saga pattern's value is invisible from the API surface; the audit brief's question "does the API actually use sagas correctly for multi-step ops?" answers as: the API delegates to core, and core's saga handles the multi-step write. Lock endpoints (`routes/records.ts:1308-1456`) are an obvious candidate for saga-style compensation but are implemented as plain async calls.

DI container usage is uneven. `routes/uuid-storage.ts:117-189` does it correctly — it pulls the storage service and config manager from `civicPress.getService('storage')`, with explicit `ServiceNotFoundError` handling, lazy `initializeStorageService(...)`, and a sane fallback message. By contrast, `routes/templates.ts:29-42` and `routes/search.ts:92-94` allocate fresh service instances per request, and `routes/info.ts:30-45` re-allocates the entire core. Three different DI patterns coexist in the same module. The `geography` router takes an already-constructed `GeographyManager` from `index.ts:218` (good); the `records` router takes an injected `RecordsService` (also good); but the rest of the route tree improvises. There is no documented module convention for how a route should reach a core service. Auth wiring is also fragile: `routes/validation.ts` and `routes/status.ts` are mounted without their required upstream middleware (see Findings VAL-002 and STAT-003), so their architectural intent (auth at route level) is silently broken by the mount-time wiring.

Internal layering follows the typical Express pattern: `index.ts` → middleware → router → service → core manager. There is no clear handler layer (the `src/handlers/` folder contains only one file, `storage-handlers.ts`, used solely by `uuid-storage.ts`); the handler/service split is essentially abandoned. Response shape is *mostly* unified through `utils/api-logger.ts::sendSuccess`, but **20 places** call `res.json(...)` directly (notably `routes/cache.ts` with 4 such calls, `routes/config.ts` with 11, and `routes/info.ts` with 1), each constructing their own `{ success: true, data: ... }` envelope by hand. A consumer building a generic error handler against the API will hit edge cases where the envelope is slightly different (e.g. `routes/audit.ts:54-63` returns `data: { entries, pagination }` directly; `routes/info.ts` returns `{ success, organization, analytics, ... }` with top-level fields rather than under `data`).

Cross-module dependencies are reasonable in declaration but loose in practice. `simple-git` (`routes/diff.ts:3`), `gray-matter` (`routes/validation.ts:21`), `nodemailer` and `@sendgrid/mail` (`routes/notifications.ts:82, 102`) are imported from npm but **not declared in `modules/api/package.json`**. They resolve today because pnpm hoists them via `@civicpress/core` (which depends on them transitively) and via workspace-wide hoisting; a clean install in an isolated environment, or a future pnpm hoist policy change, will break the API at runtime. This is a manifesto-Trust risk dressed as a packaging hygiene issue.

## Roadmap Alignment

The roadmap (`docs/roadmap.md`) marks v0.2.x **complete** with "Stable API responses" as a delivered item. The audit finds the response *envelope* mostly stable (the `success/data/error/requestId/timestamp/path/method` shape from `error-handler.ts:202-217` is comprehensive) but the *behaviour* of several endpoints (validation, status, info) does not match the level of stability the roadmap claims. The four stub routes (workflows, hooks, export, import) are roadmapped: v0.4.x explicitly names "Enhance workflow UI integration", "Improve role-based permissions UI and management", "Add comprehensive audit logs for actions", "Enhance hooks/events system" — but the API surface for those features already exists as live-looking-but-empty endpoints. That's a roadmap-vs-reality gap that misleads anyone integrating against `/api/v1/workflows` or `/api/v1/hooks` today; they will get `200 OK` and empty arrays, not informative `501 Not Implemented`.

`project-status.md` claims "API: 200+ tests passing, 90% coverage" — the workspace has 18 supertest integration files in `tests/api/` plus 3 in `modules/api/src/`. That matches the 200+ test claim by tests, not test files. Coverage at 90% across 26 route files is plausible for happy-path coverage but unverified at the handler level; routes like `templates.ts`, `geography.ts`, `uuid-storage.ts`, `indexing.ts` have no internal unit tests, so the 90% number very likely measures integration-test code-paths, not edge-case branch coverage. v0.9 ("Security review (authentication, access control, file handling)") will need to surface the validation/status/info wiring bugs and the dependency hoisting issue before the API can be called production-candidate. v1.0 ("Public API stability guarantees") will need the OpenAPI spec reconciled with the live config-driven types — currently the spec advertises an enum that no longer exists.

## Notes / Open questions

- The README at `modules/api/README.md:529-540` documents a role/permission matrix (`clerk`, `council`, `public`) that does not match the auth-service-driven config system described in `project-status.md`. Worth a doc-vs-code reconciliation pass.
- The four stub routers (workflows/hooks/export/import) raise a strategic question: are they placeholder UX so the UI can build against the surface, or are they accidents?
