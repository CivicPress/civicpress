## Technical Quality

**Code organization.** The module is split sensibly at the top level: `models/` (DB rows), `services/` (logic), `api/` (HTTP), `websocket/` (protocol + handlers), `rooms/` (device room), `realtime/` (handler glue), `workflows/` (hook handlers), `middleware/` (rate-limit). But the seams between layers leak: the `DeviceCommandService` lazily resolves `RoomManager` via the DI container (`device-command-service.ts:110–146`) because realtime initializes after broadcast-box; the `DeviceRoom` reaches through `(this.server as any).connections`, `(this.server as any).clientToDevice`, `(this.server as any).deviceConnectionMetadata` to do its work (`rooms/device-room.ts:103–148, 278–304`); `broadcast-box-services.ts` calls FOUR optional realtime-server setter methods (`setDeviceAuthDependencies`, `setDeviceCommandService`, `setDeviceConnectionTracker`, `registerRoomTypeHandler`) and additionally registers a room handler — dual-pattern wiring that no longer matches a single architecture. This is the strongest signal of unplanned-scope growth.

**Types.** Type definitions in `types/index.ts` are extensive (~530 lines) and mostly accurate, but they encode the legacy/new duality of the contract (e.g., `videoSources: string[]` and `videoSourceObjects?: SourceInfo[]`, `defaultVideoSource?: string` and `defaultVideoSourceId?: number`, `PiPConfig` and `PiPConfiguration`). The comments say things like "Legacy: String arrays for backward compatibility" but no migration path is defined. The result is N callers each picking which field to use — see UI page `index.vue` lines 184–210 where five different capability paths are merged from WebSocket status. This is the scope-creep tax.

**Errors.** A `BroadcastBoxError` hierarchy exists (`types/errors.ts`, referenced throughout) with structured codes — this is the better part. Server-side `DeviceCommandService.executeCommand` (lines 257–337) catches errors, structures them, persists a `command.error` event, and returns a typed `CommandResponse`. ACK error codes flow through `inferErrorCode` (text-matching as fallback) — fragile but documented in the integration doc. The API layer is less consistent: `api/devices.ts` register endpoint hand-codes status-code/code-string mapping (lines 504–531) rather than using the error hierarchy.

**Tests.** The README claims "78 tests passing"; current count is ~92 across 10 files. Critical gaps from the January 2025 prior audit ARE STILL PRESENT, ~4 months later:

- ❌ No `rate-limiter.test.ts` — flagged HIGH (security-critical) in prior audit.
- ❌ No `device-command-service.test.ts` — flagged HIGH (core functionality, the unified command layer) in prior audit.
- ❌ No `enrollment-cleanup.test.ts` — flagged MEDIUM in prior audit.
- ❌ No `enrollment-code.test.ts` — flagged MEDIUM (security model) in prior audit.
- ❌ `api.devices.test.ts` is **fake**: every test only asserts `expect(...).toBeDefined()` / `toBeInstanceOf(Function)` against a mock (lines 93–143). No router is ever exercised, no HTTP request is ever made. This was flagged in the prior audit as "minimal coverage"; the response was apparently to leave the file untouched. See broadcast-box-004.
- ❌ No tests for the WIP `findByCode()` method (new bcrypt-loop scan over unused codes), and no tests for the WIP `findByCode` → `findByDeviceUuid` fallback path in `registerDevice` — i.e., the WIP changes are not covered.

**Documentation.** The README is stale (still says "Phase 8 Complete", lists "Phase 9: Documentation" as next step, dated 2025-01-30). The prior audit's CLEANUP-AND-TEST-AUDIT-REPORT (~822 lines, very thorough) was created and then **deleted in commit 1181c17** ("docs: remove obsolete analysis docs and fix stale counts") on the `broadcast-box` branch — but its recommended cleanup actions are still mostly not done. Documentation has been _erased_ rather than _enacted_. The user's new `docs/broadcast-box-integration.md` (1638 lines, WIP commit) is genuinely excellent — clear, self-contained, with 21-command summary table, 5 device-lifecycle states, and a complete error-code table. But it lives alongside an older, contradictory hardware-side `docs/civicpress-integration-protocol.md` in the hardware repo (1626 lines) that describes a different on-the-wire format.

## Security (LIGHT)

**Auth on REST endpoints — partially solved, but ugly.** Every route handler in `api/devices.ts`, `api/sessions.ts`, `api/uploads.ts` still has `// TODO: Add authMiddleware from @civicpress/api` and `// TODO: Add permission check for broadcast-box:...` comments — 19 such TODOs in `devices.ts` alone, 10 in `uploads.ts`, 9 in `sessions.ts`. **However**, `registerBroadcastBoxRoutes()` (`api/index.ts`) DOES apply auth at the route-mount level: `app.use('/api/v1/broadcast-box/devices', authMiddleware, devicesRouter)` (line 237) with a separate public router for `POST /` (device self-enrollment, gated by rate-limiter). So in practice the endpoints ARE protected — but a future maintainer reading the route file would not know that. The TODO comments are now actively misleading. Permission-level RBAC (`broadcast-box:devices:list`, `:view`, `:update`, `:control`, etc.) is still entirely absent: a user with any valid JWT can list, view, update, revoke, and command any device.

**Rate limiter — security gate gets switched off in non-prod.** WIP commit `47d0ff6` added a `process.env.NODE_ENV !== 'production'` short-circuit in `middleware/rate-limiter.ts` (lines 170–174). Combined with the WIP `findByCode()` change (bcrypt-loop over up to 24 hours of unused codes), an unauthenticated `POST /api/v1/broadcast-box/devices` request will run N bcrypt compares per attempt in non-production. With bcrypt at 12 rounds (~250ms/compare) and 100 unused codes, a single registration attempt costs ~25 CPU-seconds — and there is no rate limit. This is a denial-of-service vector in any non-prod CI/test/dev/demo environment. See broadcast-box-007.

**Enrollment-code re-registration recovery path is permissive.** `DeviceManager.registerDevice` (lines 170–264) allows re-registration of an existing device using an expired or already-used enrollment code, as long as the bcrypt hash matches. The intent (recovery from token loss) is reasonable, but it means: an attacker who once obtained the cleartext enrollment code (e.g., from a QR-code screenshot, clerk's notebook) can forever re-register the device and get a fresh long-lived token. There is no rotation mechanism, no "rotate enrollment secret on activation" step. Codes never expire from this path. See broadcast-box-008.

**Admin endpoints have a "dev mode = no auth" branch.** `api/index.ts` lines 91–103: if `authMiddleware` is not provided, the admin endpoint `POST /api/v1/broadcast-box/admin/reset-rate-limits` is registered with NO auth and a warning log. This is a fingerprintable foot-gun if the wiring ever forgets to pass `authMiddleware` (test setups, partial init paths, the optional-module dynamic-import flow at `modules/api/src/index.ts` lines 258–339).

**Token storage and validation.** Device tokens are JWT, issued by `DeviceAuthService.generateToken`, encrypted at rest on the device with Fernet (per hardware repo). Server accepts the token via query param, `Authorization: Bearer`, or `X-Device-Token` header — three places (per integration doc § 9). The query-param form is convenient but leaks tokens into access logs and proxy histories; this would warrant a dedicated security session.

**Debug logging leaks request bodies and command IDs.** `modules/api/src/index.ts` lines 76–106 register a middleware that `console.log`s the raw body of every `POST /broadcast-box/devices/enroll` request — including the cleartext enrollment code if present. `api/devices.ts` lines 260–276 logs the full validation request body to console.error on every validation pass/fail. `DeviceCommandService.sendCommandViaWebSocket` lines 374–381 logs the entire command (including payload) to coreInfo on every send. In production this is at minimum a noisy log surface and at worst a leak of stream keys (`stream.configure` payload includes `stream_key`). See broadcast-box-011.

## AI-Generation Smells

**`protocol-adapter.ts` (172 lines) is exported but never imported anywhere.** It defines six bidirectional adapters between two message formats ("protocol" snake_case nested under payload, vs. "current" camelCase flat). The production code path uses neither — `device-room-handler.ts` lines 422–475 contains a 50-line hand-rolled ACK normalizer that does essentially the same job but inline. This is classic AI-generated scaffolding: a clean abstraction was built, then drift left it stranded. (Found via `rg "adaptProtocol|adaptCurrent|protocol-adapter"` across all module sources — zero matches.)

**Unused server-side command handlers.** `websocket/command-handlers.ts` (1124 lines) builds a full `CommandHandlerRegistry` with handlers for `start_session`, `stop_session`, `update_config`, etc. — but the actual flow is CivicPress → Device (commands are SENT, not RECEIVED, by the server). The realtime server does not feed device-sent "commands" back to this registry. These handlers appear unreachable in the current device-room handler flow; they exist for a different architecture (perhaps an earlier design where the server received commands from external schedulers via the device-room WebSocket). The 12 command-handler tests pass because they exercise the registry directly, but the registry is not wired into the runtime. Distinguish from intentional dead code: there's no comment explaining the divergence. See broadcast-box-013.

**Massive composables with manual mount-tracking.** `useDevicePreview` (1419 lines), `useDeviceConnectionStatus` (1459 lines), `useDeviceCommands` (969 lines) — these are not "composables" in the Vue idiomatic sense (a small reactive primitive); they are entire stateful machines that re-implement event-handler registration, subscription bookkeeping, and lifecycle hooks. The device page `[id]/index.vue` has manual variables `isComponentMounted`, `lastSubscribedUuid`, `subscriptionTime`, `lastConnectedValue`, `hasRefreshedAfterCapabilities`, `lastRefreshedDeviceUuid` (lines 85–158) plus 4 `watch()` blocks managing them. The repeated pattern `if (!isComponentMounted) return;` suggests the composables don't own their own unmount cleanup — the page is babysitting them. This is the scope-creep tax made concrete.

**Console.log in production paths.** Found at minimum:

- `modules/api/src/index.ts` lines 77–106: raw body logger for broadcast-box enroll requests.
- `modules/broadcast-box/src/api/devices.ts` lines 260, 263–266, 275, 286–291, 384–394: validation middleware and the enroll handler dump the request body to console on every call.
- `modules/broadcast-box/src/rooms/device-room.ts` lines 172–176, 197–204, 225–239, 262–272, 344–354, 396–413, 440–453, 471–484, 488–494, 500–521: every command-send attempt logs to console.log/error with the full message JSON, connection score, ready states.
- `modules/broadcast-box/src/services/broadcast-box-services.ts` lines 501–502, 586–588: registration logs to console.log on top of structured logger.
- `modules/broadcast-box/src/services/device-command-service.ts` lines 374–381: every command pending-store logs to coreInfo with `commandFull: JSON.stringify(command)`.

These look like debugging instrumentation that was added during integration battles and never removed.

**Defensive over-coding around DI resolution.** `broadcast-box-services.ts` (752 lines) is dominated by nested try/catch blocks resolving `realtimeServer` four different times, each with its own `SERVICE_NOT_FOUND` swallow-and-warn. The pattern reads as Claude/Cursor defensive boilerplate accreted by repeated "service might not be available — let's just try" prompts. A single dependency-check + a single mounting function would be ~150 lines.

**`console.log('=== ... ===')` debug banners (e.g., `=== ENROLL ENDPOINT CALLED ===`, `=== BROADCAST BOX ENROLL ERROR ===`) in route handlers.** Classic AI shotgun-debugging that should never have been committed.

**Test setup over-engineering vs. test assertion under-engineering.** `api.devices.test.ts` (146 lines) has elaborate mocks for `deviceManager`, `deviceAuth`, `connectionTracker`, `mockLogger`, beforeEach setup, etc. — and the actual assertions don't exercise any of them. This is the AI signature of "make it look like there are tests" without actually testing.

## Architecture

**Module boundary leakage.** The realtime server has FOUR setter-injection points specifically for broadcast-box (`setDeviceAuthDependencies`, `setDeviceCommandService`, `setDeviceConnectionTracker`, `registerRoomTypeHandler`), AND `broadcast-box/rooms/device-room.ts` reaches through `(server as any).connections`, `(server as any).clientToDevice`, `(server as any).deviceConnectionMetadata`. Two patterns coexist: legacy setter injection AND the newer "registerRoomTypeHandler" pattern, and the code uses BOTH simultaneously (`broadcast-box-services.ts` lines 496–750). This means `@civicpress/realtime` is no longer a generic Yjs collab module — it knows about device auth, device commands, and device connection tracking. That's a directional architectural mistake: either broadcast-box owns its own WebSocket server, or realtime owns a clean room-type-handler API that broadcast-box plugs into. The half-and-half state is the most concrete answer to the user's central question.

**Optional-module dynamic imports.** `modules/api/src/index.ts` lines 257–339 attempts THREE different ways to import `@civicpress/broadcast-box`: by package name, by `src` path (dev), by `dist` path (prod), each with its own catch/log. `broadcast-box-services.ts` uses similar conditional path-juggling for migration files (lines 92–134). The contortions are because the broadcast-box module sits in a workspace package and may or may not be linked. This is symptomatic of treating broadcast-box as both an internal module AND an optional plugin. Pick one.

**Service registration order is fragile.** `core/src/civic-core-services.ts` lines 484–494 note "Realtime server registered (will initialize after broadcast-box services)" — initialization is intentionally deferred so broadcast-box can inject its dependencies first. This works but is implicit; nothing enforces the order. The `DeviceCommandService.getRoomManager()` lazy-resolution wrapper (`device-command-service.ts` lines 110–146) exists specifically because of this ordering issue and logs "CRITICAL: DeviceCommandService requires RoomManager…" if the order breaks. That's a load-bearing comment in a runtime path.

**Contract surface (the user's specific question).** Three distinct layers must agree:

1. **REST enrollment** (`POST /api/v1/broadcast-box/devices`): clear, well-validated, rate-limited (in prod), with structured error codes.
2. **WebSocket auth & lifecycle** (token in query/header, `device.connected` event on connect, `connection.ack`/`error` from server, auto-re-enrollment on `AUTH_FAILED`): mostly clear, documented in the new integration doc § 3 and § 9. The 5-state device lifecycle (ENROLLED → ACTIVE → SUSPENDED → REVOKED → DECOMMISSIONED) is named but **never actually transitioned in code** — `SUSPENDED`/`DECOMMISSIONED` are defined in `types/index.ts:28-33` but no service method moves a device to or out of them.
3. **Command/ACK on-the-wire format**: this is where the contract IS ambiguous. The hardware-side `civicpress-integration-protocol.md` § "Commands" describes `{ type: "command", payload: { action, ... }, id, timestamp }` — action INSIDE payload, snake_case. The server-side `protocol.ts:174-186` creates `{ type: "command", id, timestamp, action, payload }` — action OUTSIDE payload, camelCase. The server-side integration doc (the new WIP one) § 4 acknowledges THREE accepted formats and says the device normalizes internally. ACK direction has the same split: device returns `{ type: "ack", payload: { command_id, success, result } }`; server expects `{ type: "ack", commandId, success, error, payload }`. The translation is patched ad-hoc in `device-room-handler.ts:422–475`. This is what the user means by "not always clear" — the contract HAS been documented (well, in the WIP integration doc) but the code path does not have a single canonical translator. The unused `protocol-adapter.ts` was meant to be it. See broadcast-box-010.

**Sources state has three sources of truth.** Device capabilities arrive via (a) the initial REST registration `capabilities` field, (b) the WebSocket `device.connected` event payload, (c) periodic `status` messages. The server merges all three into the `BroadcastDevice.capabilities` field, but each has slightly different shape and the merge is hand-coded across `event-handlers.ts` (lines 178–520 for `device.connected`, lines 906–1450 for `status`). The UI then re-merges in `index.vue` (lines 161–254). Three merge sites for the same field. See broadcast-box-014.

## Roadmap Alignment

**Reality check.** `docs/project-status.md` and `docs/roadmap.md` mention broadcast-box **zero times**. The manifesto (`manifesto/manifesto.md` §3.5) still names Ledger as the flagship for civic accountability. The README inside the module is stamped "Phase 8 Complete" with `**Last Updated**: 2025-01-30` and lists Phase 9 (Documentation) and Phase 10 (Permissions & Security) as "Next Steps" — both still not done ~4 months later in audit time. The pattern is: internal docs have been actively maintained (status protocol integration, PiP commands, integration doc, project state); top-level docs have not. As the user noted, this is known and a post-audit cleanup target — it should land as part of the same cycle that publishes the audit, because the gap itself reads as a Transparency-principle violation per manifesto §"Transparency": "Decisions, records, and changes should be visible by default."

**Milestones threatened.** broadcast-box is not on the roadmap, so technically nothing is "threatened" by the findings below. But if broadcast-box is to ship as the flagship (as the strategic shift implies), several findings below must land before any municipal pilot (v0.5–0.8 per roadmap): (a) the Markdown-civic-artifact gap (`broadcast-box-002`), (b) the live-event setup UX gap (`broadcast-box-005`), (c) the contract ambiguity (`broadcast-box-010`), (d) RBAC permissions (`broadcast-box-006`), (e) the dev-mode-no-auth foot-guns (`broadcast-box-007`, `broadcast-box-011`).

**The big architectural question the user asked.** _Is the current implementation approach correct?_ The honest answer is: **the approach is roughly right but the seams are wrong**. The right thing was: (1) treat the device as a passive appliance with an explicit protocol, (2) keep recording resilience on the device side, (3) attach recordings to civic session records on the server. The wrong things — accumulated as the scope grew — are: (1) the realtime server knows too much about devices; (2) the contract has three formats and no canonical translator; (3) state syncs through three layers (REST + connection event + status messages) with hand-coded merges in each; (4) commands flow through a "unified" DeviceCommandService that itself reaches lazily into the DI container for a RoomManager that may not exist; (5) the recording pipeline produces blobs, not civic artifacts. **Recommendation: an architectural refactor, not a cleanup.** Specifically — define a single Device Contract package that owns ProtocolHandler + ProtocolAdapter + DeviceRoom + DeviceCommandService, expose it to realtime via a single RoomTypeHandler entry-point, retire the four setter-injection methods on realtime, retire the unused command-handler registry, and add a "civic-artifact derivation" service that produces session-record updates from recording metadata.

## Notes / Open questions

- **Hardware-side audit (target 10) will confirm the contract picture.** This audit reads the contract from the server side; the hardware-repo audit reads it from the device side. Synthesis in Phase 2 should specifically reconcile the two and tag the canonical command/event/ack format that BOTH sides will commit to.
- **The user's WIP integration doc (`docs/broadcast-box-integration.md`, 1638 lines, untracked → added in commit 47d0ff6) is the best piece of documentation in the module.** It should be promoted to `modules/broadcast-box/docs/integration.md` and the hardware repo's older `civicpress-integration-protocol.md` should be reconciled against it (probably the hardware one is the stale fork — but a human needs to decide).
- **Civic-artifact derivation is the manifesto's biggest hidden win.** A small derivation service that emits a Markdown minutes scaffold (timestamp + active source + recording duration per state transition, with placeholder attendee/motion sections) would close the §3.3 gap and make broadcast-box a real flagship for §3.4 "Public by Default". This is L effort but transformative.
- **The deleted `CLEANUP-AND-TEST-AUDIT-REPORT.md` (822 lines) is the proximate prior art for this audit.** Most of its findings (rate-limiter tests, device-command-service tests, enrollment-code tests, the fake api.devices test) are still open. Phase 2 should include a checklist that explicitly tracks "from Jan 2025 audit → status" so we don't audit-and-forget twice in a row.
- **The WIP commit `47d0ff6` itself is small and reasonable as a checkpoint** (auth-status-code fix, dev-mode rate-limit skip, `findByCode` recovery path, status code change 401 → 422 for invalid-credentials). The substantive in-flight work is in the UI page redesign — and the architectural refactor recommended above would probably reset that redesign onto a smaller, more stable composable footprint.
- **Roadmap alignment recommendation:** as part of the post-audit cleanup, add a v0.3.x "Broadcast Box — flagship" line item to `docs/roadmap.md`, update `docs/project-status.md`, and either revise manifesto §3.5 to acknowledge broadcast-box alongside Ledger or — better — name the broader principle ("civic record capture and accountability") and let both Ledger and broadcast-box be named as flagship modules under it.
