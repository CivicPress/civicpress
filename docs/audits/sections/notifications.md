## Technical Quality

The implementation is structurally split across two locations in a way that does not serve the codebase. **`modules/notifications/`** is a 1-file folder containing an `EmailChannel` class that **nothing in the runtime actually uses**: the CLI (`cli/src/commands/notify.ts:50`), the API (`modules/api/src/routes/notifications.ts:64`), and the auth flow (`core/src/auth/email-validation-service.ts:105-192`) each define their own inline `EmailChannel`-like object literal. The module file is referenced **only by `tests/core/notification-system.test.ts:12`**. It is, in effect, a museum piece that the tests confirm "works" while production code routes around it. Worse, the module's `EmailChannel.sendViaSMTP` and `sendViaNodemailer` (`email-channel.ts:129, 188`) call `nodemailer.createTransporter(…)` — a method that **does not exist** in nodemailer (correct API: `createTransport`) — so both code paths would throw at runtime. The tests never exercise them because the tests stop at `validateConfig()` / `getCapabilities()` (`tests/core/notification-system.test.ts:201-232` literally comments "we'll test that the channel is properly configured… can handle the request structure" instead of actually sending).

The core layer (`core/src/notifications/`) is more substantial — ~1,800 LoC across `NotificationService`, `NotificationConfig`, `NotificationAudit`, `NotificationQueue`, `NotificationRateLimiter`, `NotificationSecurity`, `NotificationLogger`, `NotificationTemplate`, plus an `AuthTemplate` subclass. But several of these classes are **instantiated but never used**: `NotificationService` constructs a `NotificationQueue` in its constructor (`notification-service.ts:46`) but never calls `enqueue` or `processQueue` from anywhere in the codebase; `NotificationLogger` accumulates an in-memory array of log entries that nobody reads. Methods like `NotificationSecurity.encryptSensitiveData`, `decryptSensitiveData`, `generateSecureToken`, and the entire `validateWebhookSignature` / `generateWebhookSignature` pair are exposed but unused outside `auth/email-validation-service.ts`.

Test coverage is thin and misleading. The only test file (`tests/core/notification-system.test.ts`) tests **only** the email channel, against a `nodemailer` config pointed at `smtp.test.com:587` (non-resolvable). The headline "should register and use email channel" test (line 57) sends a request and asserts `result.success === true` — but the production code's `success` is computed as `sentChannels.length > 0`, and the test fixture's `nodemailer` host is fake, so this passes only because the channel registration succeeded, not because mail was sent. SMS, Slack, and webhook channels are **never tested**.

The `NotificationConfig.loadConfig` swallows yaml parse errors silently (`notification-config.ts:125-128`) and falls back to defaults, masking misconfiguration. Type safety is loose: `getChannelConfig(name: string): any` discards type info; the API route route uses `as any` casts throughout (`routes/notifications.ts:119, 122`). The `priority` and `scheduledAt` / `expiresAt` fields on `NotificationRequest` are defined but never honored anywhere in the send path.

## Security (LIGHT)

**Critical issue: gates are inert.** `NotificationService.sendNotification()` calls `await this.security.validateRequest(request)` (`notification-service.ts:108`) and `await this.rateLimiter.checkRateLimit(request)` (`:111`) but never inspects either return value. `validateRequest` returns `{ valid, errors, warnings }` without throwing; `checkRateLimit` returns `{ allowed, remaining, resetTime }` without throwing. The early-return logic that *should* live here is absent, so:
- Requests with more than 10 channels, oversized data, missing required fields → still proceed.
- Requests over the per-channel hourly rate limit → still proceed; the only side effect is that the rate-limiter's in-memory counter increments.
- Requests containing `<script>` tags trigger only a `warning` in the validator, then proceed.

**PII filtering is broken in two ways.** (1) The PII regexes (`notification-security.ts:11-16`) include `\b\d{10,11}\b` which will match any 10–11-digit run (ZIPs+suffixes, ID numbers, dates concatenated as digits) and `[A-Z|a-z]` in the email pattern is a literal pipe inside a character class, not alternation — minor but indicative of unreviewed AI output. (2) `sanitizeContent` is applied to `request.data` (`notification-service.ts:124`) which **is** the template variable bag. So if a verification email uses the recipient's email address as a template variable, the variable gets replaced with `[REDACTED]` *before rendering* — the recipient receives an email saying "Hello [REDACTED], please verify". Spot the design confusion: the system tries to redact PII from the audit log but applies the redaction to the message body instead.

**Citizen PII in audit log.** The audit log writes only `{channels, success}` (`notification-service.ts:153-160`); it does *not* write the recipient. That's actually a positive for PII minimization — but it means the audit is useless for compliance/transparency questions ("did this user get notified about X?"). The opposite trade-off is happening in the API route at `routes/notifications.ts:141-142`, which logs `metadata: { provider, to }` to the general `AuditLogger` — i.e. the citizen's email **is** stored in the API audit trail, just not in the notifications-specific one. Inconsistent.

**Webhook security exists in code but is unwired.** `NotificationSecurity.validateWebhookSignature` (`:184`) and `generateWebhookSignature` (`:210`) are well-implemented (HMAC-SHA256, reject when secrets manager not initialized) but **no inbound webhook endpoint and no outbound webhook channel exist**. So the security primitives are correct but defend against nothing. The spec (`docs/specs/notifications.md:75`) declares "All webhook endpoints must be cryptographically signed with HMAC-SHA256" — that promise is unenforceable because there are no webhooks.

**API authorization.** The `POST /api/v1/notifications/test` route does `router.use(requirePermission('system:admin'))` (`routes/notifications.ts:14`), so the test endpoint is admin-gated. Good. No issues there.

**Recommendation:** A dedicated security session should fix the inert validation/rate-limit checks (Critical), redesign the PII sanitization (Critical), reconcile what is actually logged where, and decide whether to delete the unused webhook primitives or wire them.

## AI-Generation Smells

Strong indicators of AI-assisted scaffolding throughout.

**Fake-comprehensive provider menu.** `EmailChannel` in `modules/notifications/channels/email-channel.ts` enumerates four providers (`sendgrid`, `ses`, `smtp`, `nodemailer`) and the config schema (`notification-config.ts:7-50`) does too. But `ses` is `// TODO: Implement AWS SES integration` followed by 20 lines of commented-out code (`email-channel.ts:88-112`) and immediately throws `'AWS SES integration not yet implemented'`. The CLI's parallel `EmailChannel` only implements `smtp` and `sendgrid`. So three of four advertised options are partial or absent. This is exactly the "fake comprehensiveness" smell.

**SMS and Slack channels: declared, never written.** Config schema, rate limits, and `getChannelRecipient`'s switch case all reference `'sms'` and `'slack'` (`notification-service.ts:238-241`, `notification-rate-limiter.ts:88-93`), but no `SmsChannel` or `SlackChannel` class exists in the repo. The spec lists them as supported channels. The settings page exposes config fields. Nothing sends.

**`simulateEmailSending` private method, never called.** `email-channel.ts:386-394` defines a `private async simulateEmailSending` with a 100ms timer and a 5% random failure — clearly a Cursor-style "let's leave a development helper". Dead code.

**The `NotificationQueue` paradox.** A whole class with `enqueue`, `processQueue`, exponential backoff, `getQueueStats`, `clearQueue` — and **nobody calls any of it**. Constructed in `NotificationService` for no reason. The CLI `notify:retry` command (`cli/src/commands/notify.ts:540-600`) actually says `cliWarn('Retry functionality not yet implemented', 'notify:retry')` — i.e. the command is shipped, the queue class exists, they never met.

**Four ad-hoc reimplementations of EmailChannel.** Same pattern of dynamic `import('nodemailer')` followed by `createTransport({...credentials})` followed by `verify()` and `sendMail` appears in `cli/src/commands/notify.ts:50-184`, `modules/api/src/routes/notifications.ts:64-115`, `core/src/auth/email-validation-service.ts:117-183`, and `modules/notifications/channels/email-channel.ts:118-219`. They all do roughly the same thing; none of them reuse the others. The duplication wouldn't be remarkable in a hobby project, but for the "extensible notification system" the spec describes, it's textbook AI sprawl.

**Comments that explain WHAT not WHY.** Examples: `// Reset rate limit` (`notification-rate-limiter.ts:59`), `// Process results` (`notification-service.ts:137`), `// Remove from queue on success` (`notification-queue.ts:56`), `// Send to each channel` (`notification-service.ts:126`). Six dozen of these. None explains a non-obvious civic decision.

**JSDoc on every method.** Every public method, including one-liners like `getName(): string { return this.name; }`, gets a multi-line JSDoc block. The codebase as a whole has lighter comments — this folder is conspicuously over-commented.

**Inconsistent error handling.** `NotificationSecurity` throws when secrets manager is missing (`:212`); `NotificationConfig` silently falls back to defaults (`:125-128`); `NotificationAudit` calls `coreError` and swallows the error (`:84-93`); `NotificationService.sendNotification` re-throws after logging (`:175-190`). No domain-error hierarchy is used, despite the rest of `core/src/errors/` having one.

## Architecture

The notifications stack sits in an awkward in-between place between "module" and "core service". The `modules/notifications/` directory was apparently meant to host channel implementations (channels/email-channel.ts) while orchestration lived in core — a reasonable split — but the split never landed: the CLI, API, and auth-service each inlined their own channel because the published one had bugs (`createTransporter` typo) or didn't normalize the metadata-wrapped YAML config (`{value, type, …}`) that the rest of the system uses.

DI integration is partial. `core/src/civic-core-services.ts:139` registers `NotificationService` in the DI container, and `core/src/civic-core.ts:346` exposes a `getNotificationService()` accessor. But the CLI (`commands/notify.ts:228, 231`), API route (`routes/notifications.ts:118`), and auth-service (`email-validation-service.ts:43-44`) all instantiate `new NotificationService(new NotificationConfig())` directly, bypassing DI. That means there are at minimum four parallel `NotificationService` instances at runtime — each with its own in-memory rate-limit counters and in-memory queue. Per-channel hourly limits are therefore not truly per-channel — they are per-service-instance.

The boundary between `core/notifications` and `modules/notifications` is not justified by any architectural principle in the project — the `core/notifications` code is not generic infrastructure, and the `modules/notifications` code is not module-specific. Compare with `modules/storage/`, which has a clean providers/ abstraction and a README; `modules/notifications/` looks like an aborted refactor.

Cross-module dependencies are weak: nothing in `modules/api/` or `modules/ui/` imports anything from `modules/notifications/`. The only import of `modules/notifications/channels/email-channel.ts` in the entire repo is the test file. This means deleting `modules/notifications/` would have zero runtime impact and only one test would need updating. That alone tells you the module is dead-on-arrival.

The Hooks integration the docs promise (`docs/notifications.md:274-282` — `event: 'user:registered' → action: 'notify:email_verification'`) is not wired. `auth-service.ts` calls `EmailValidationService` directly; no hook system routes events into notifications. The spec's "Trigger notifications on system events (hook:-based)" responsibility is not met.

## Roadmap Alignment

`docs/roadmap.md` does not list notifications as a deliverable in any phase. `docs/project-status.md` mentions notifications only as a Settings UI page that exists. `docs/specs/notifications.md` declares status: **stable v1.0.0** — flatly contradicting reality (SMS not implemented, webhooks not implemented, queue not wired, hooks not wired, audit log structurally false). The spec was written 2025-07 and never reconciled with implementation. This is a Transparency-principle harm: a "stable" spec that the code does not satisfy erodes the documentation's credibility.

For roadmap milestones: notifications are a latent prerequisite for **v0.4.x — Workflow Engine + Permissions** (workflows that emit events need a notification target), **v0.5–0.8 — Municipal Pilot Readiness** (clerks need to be alerted to pending approvals; citizens need confirmation of submissions), and **v1.0** (proven deployments imply reliable transactional email for password resets at minimum). The current state — broken module file, four inline copies, inert rate-limit/validation gates, untruthful audit log — would block any honest pilot deployment. Fixing notifications is on the path to v0.5; it has not been planned.

## Notes / Open questions

- The `modules/README.md` (at the modules folder root) is mis-titled "CivicPress UI Module" and describes the UI module — i.e. it is *not* a modules-index. This isn't a notifications-specific issue but it compounds the discoverability problem; it's worth a separate `modules/README.md` fix during the audit cleanup.
- The `core/src/notifications/` location for orchestration code is itself questionable. Either the folder name `modules/notifications/` should be retired (orchestration AND channels in core) or the channel classes should be moved into `modules/notifications/` properly with a real module shape. The current split looks like an unfinished refactor.
- The `NotificationService` is registered in the DI container (`civic-core-services.ts:139`) but every caller bypasses it. Recommend either (a) deleting the DI registration to remove the impression of consistency, or (b) refactoring callers to resolve via `container.resolve('notification')`.
- Whether the audit log should include `recipient` is a real privacy trade-off worth a human decision. Including it makes the log compliance-useful but also makes the log a citizen-PII store. The current behavior — omitting recipient from the notifications log while logging `{provider, to}` to the *general* `AuditLogger` from `routes/notifications.ts:141` — is an accidental compromise, not a deliberate one. Worth designing.
- The spec `docs/specs/notifications.md` should likely be moved out of "stable v1.0.0" status until the implementation catches up. This is a Phase 2 synthesis-level recommendation but tied to finding notifications-007.
- Mention this in the synthesis: notifications is a prerequisite that has slipped under the radar because no roadmap milestone names it. A small per-recipient rate-limit + a real audit-log truthfulness fix + a single canonical EmailChannel is probably 1–2 sprints of work and would materially derisk v0.5–0.8 pilot readiness.
