## Technical Quality

The module's surface area is reasonable on paper — eight source files for transports, rooms, snapshots, presence, auth, errors, config, and a handler-registry — but **the actual code is dominated by one monolithic file**: `src/realtime-server.ts` is 3,581 lines, more than 2× the next largest file. Within that file, **roughly 1,500 lines** (rough estimate from the device routing block at line 551 and the legacy device message handler at line 1533 through ~2510) are device-specific logic for broadcast-box — JSON-message handling, ack normalization across three competing payload shapes, source/capability merging, active-source enrichment, ICE/SDP relay between device and observers. The realtime module is supposed to be a generic Yjs WebSocket server; the device logic is supposed to live in `modules/broadcast-box/src/realtime/device-room-handler.ts` (a 553-line `RoomTypeHandler` implementation that was added in commit `e014f40` specifically "to decouple broadcast-box device handlers from realtime module"). Both code paths still exist in parallel, with the legacy path activated only as a fallback (`src/realtime-server.ts:530-548` — `if (customHandler) { … return; }` — otherwise legacy device branch at line 551).

Type safety is weakened by **46 occurrences of `: any`** in `realtime-server.ts` alone. Four `private` fields are explicitly typed `any | null` with comments like `// DeviceAuthService - avoiding circular dependency` (lines 140–146) and marked `@deprecated Use handler registry instead`. The setters that populate them (`setDeviceAuthDependencies`, `setDeviceCommandService`, `setDeviceConnectionTracker`) are also marked deprecated but still ship and are still wired up in `standalone-realtime.mjs` (lines 79–116). This is **transitional code that has been transitional for at least one release** — both contracts coexist and neither is removed.

Test coverage is broad but with gaps that matter for civic trust:
- `auth.test.ts` covers token extraction and URL parsing (good).
- `rate-limiting.test.ts` exercises `checkMessageRateLimit` per-client, but **does not test `checkConnectionLimits`** — the function with the `userId = null` bug.
- `realtime.integration.test.ts` (~1k lines) tests multi-client editing, sync convergence, large payloads, and reconnection — high-value end-to-end coverage.
- **No test exercises the connection-cleanup leak** (`connectionCounts` not decremented on user disconnect; `userConnections` Set entries never removed).
- **No test exercises the offline-edit / late-sync scenario** that the manifesto's resilience constraint calls out — every integration test connects, edits, and disconnects within seconds.
- The 1500-line device legacy path in `realtime-server.ts` is **not tested** at all from this module's tests; it's tested only via `modules/broadcast-box`. So if the legacy path is reached (because the handler registry isn't wired), no realtime-module test will catch a regression.

Documentation is plentiful (`README`, `ARCHITECTURE`, `DEPLOYMENT`, `DEVELOPMENT`, `TESTING`, `QUICK-START`, `STANDALONE`, `IMPLEMENTATION-STATUS`, `CHANGELOG`) but **partly stale**: `TESTING.md` and `test-websocket.mjs` still describe the JSON-message API removed in commit `5d73791` ("fix(realtime): update tests for binary y-protocols and remove dead code"). `DEPLOYMENT.md` includes a `tls.enabled` config option that is not in the `RealtimeConfig` type (`src/types/realtime.types.ts:7-32`); a deployer following the doc would silently get plain `ws://`. `IMPLEMENTATION-STATUS.md` lists "TODO: rate limiting per connection" — that's been done — but doesn't mention the connection-cleanup leak that remains.

## Security (LIGHT)

Auth is layered correctly: tokens accepted via three methods (header / subprotocol / query) in priority order with a `coreWarn` on the deprecated query method (`src/auth.ts:127-179`); device tokens include a UUID-match check between URL and token payload (`src/auth.ts:219-228`) to prevent token-reuse for a different device. Permission checks call `authService.userCan('records:view'/'records:edit', …)` and refuse the connection on `canView=false`. Drafts are looked up via an optional `databaseService.getDraft()` (`src/auth.ts:60-63`), which is necessary for collab-edit-before-publish flows.

Surfaces I'd flag for a dedicated security session:

1. **Per-user connection limit is dead code.** `checkConnectionLimits(clientIp, null)` is called at the top of `handleConnection` (`src/realtime-server.ts:472`), so the `connections_per_user` branch in `checkConnectionLimits` (`src/realtime-server.ts:2948-2960`) is unreachable. The `userConnections` map is populated on connect (`src/realtime-server.ts:3014`) but never read for limit enforcement and never cleaned up on disconnect — so even if the call were moved post-auth, the count would grow unboundedly and lock out legitimate users.

2. **Connection-count leak.** `connectionCounts` (per-IP) is decremented in the new handler-registry path (`src/realtime-server.ts:1492-1497`) but **not** in `handleDisconnect` (legacy user path, `src/realtime-server.ts:2854-2921` — note the explicit `// TODO: Track IP and user for cleanup` at line 2891) or in `handleDeviceDisconnect` (`src/realtime-server.ts:2517-2597`). Long-running deployments will hit `connections_per_ip` and start rejecting legitimate connections.

3. **The legacy device path's ack/status handling does heavy JSON manipulation with `any` types** (`src/realtime-server.ts:1571-2330`, ~750 lines). It normalizes three competing ACK shapes (`message.commandId`, `message.payload.commandId`, `message.payload.command_id`), reads back into the device-manager, and updates DB device config based on payload content. Any malformed status message from a compromised device (or a device with mismatched firmware) flows directly into a `deviceManager.updateDevice()` call. Worth a dedicated review with the broadcast-box hardware contract in mind.

4. **Snapshot data is unsigned, untyped, and unbounded.** `realtime_snapshots.snapshot_data BLOB NOT NULL` (`src/realtime-server.ts:3180-3193`). No size limit on per-snapshot or per-room storage; no integrity check; no schema for the binary payload beyond "whatever Yjs produces today". `Y.applyUpdate` on a corrupted blob can leave the document in a partially-applied state; the room then continues serving that state to clients. This is more an integrity/durability flag than a CVE.

5. **`y-websocket` (client) handles its own reconnection inside the WebSocket subprotocol** — the realtime server has no server-side resync limits. A misbehaving or hostile client can churn reconnect attempts; the per-IP limit (when it's not leaked) is the only brake.

6. **`coreInfo` / `coreDebug` calls in the device path log full token previews** (e.g. `tokenPreview: token.substring(0, 20) + '...'` at `src/realtime-server.ts:670`, `src/auth.ts:198`). 20 chars of a JWT is not enough to forge, but worth flagging for the security session.

No critical "obvious CVE" surfaces appeared. The above are configuration / liveness / data-integrity concerns, not RCE-class.

## AI-Generation Smells

- **Dead code retained "in case it's useful".** `generateParticipantColor()` and the 16-entry `PARTICIPANT_COLORS` palette (`src/realtime-server.ts:75-101`) are declared but never called anywhere in the realtime module. `CLAUDE.md` confirms this is known and intentionally kept. The exact same color array exists in `modules/ui/app/composables/useRealtimeEditor.ts:56-75` (`generateUserColor`), which **is** used. The server-side copy is fossil.
- **Dead reconnection scaffolding on the client.** `MAX_RECONNECT_ATTEMPTS = 5` and `RECONNECT_DELAYS = [1000, 2000, 4000, 8000, 16000]` in `modules/ui/app/composables/useRealtimeEditor.ts:114-116` are declared but never referenced; `reconnectAttempts` is only ever assigned `= 0` (lines 234, 300), never incremented or read against the max. Looks like a planned-then-abandoned manual reconnect path superseded by y-websocket's internal logic, but the variables were left as if they're load-bearing.
- **Two parallel device-handling paths.** `src/realtime-server.ts` contains both the new `handleConnectionWithHandler` (line 1257, generic handler registry) and the legacy `handleDeviceConnection` + `setupDeviceMessageHandlers` (lines 640, 1533, ~2000 lines). The legacy path is gated by `if (customHandler) { … return; }` (line 530) and is effectively dead in any deployment where broadcast-box registers its handler. The decoupling commit (`e014f40`) explicitly says "Decouple broadcast-box device handlers from realtime module," but only the new path landed — the old one was kept "for safety". This is a classic AI-coding pattern: add the new abstraction, leave the old one with `@deprecated` comments, never delete.
- **Three competing ACK message shapes** are normalized in a 100-line conditional cascade (`src/realtime-server.ts:1571-1665`). Comments document "Format 1 / Format 2 / Format 3 (missing type field — handle gracefully)". This is symptomatic of a contract that grew without ever being collapsed; rather than fix one side, both sides accumulate compatibility code.
- **Heavy emoji logging in production code paths.** `'📤 Observer sending preview.answer'`, `'✅ preview.answer forwarded'`, `'❌ Failed to send'`, `'📥 Device sending ICE candidate'` (`src/realtime-server.ts:2370, 2388, 2397, 2411-2412`). Fine for development, but these are `coreInfo` (info level) and will land in production logs at high volume during streaming.
- **`as any` casts as a habit.** 46 instances in one file. Several are justified by stated circular-dep avoidance (`deviceAuthService: any | null`), but many others (`getState: () => room.getState() as any`, `clientData: any`, `let configUpdates: any = {}`) appear to be friction-shedding rather than necessary.
- **Defensive logging that exceeds defensive code.** Each `coreError` / `coreWarn` call passes the same 6-field context object three times with slight variations: once as the data arg, once as the error context, once as the operation metadata. `realtime-server.ts` has ~80 such calls and they account for a large chunk of the line count without adding behavior.

## Architecture

**Boundary placement.** The intended architecture is sound: `RealtimeServer` exposes a `HandlerRegistry` (`src/handler-registry.ts`); external modules (broadcast-box) implement `RoomTypeHandler` (`src/types/handler-registry.types.ts`) and register at startup; the server only knows about records and ping/pong directly. The actual code partially realizes this: the legacy device path is still there, and the server still imports zero types from broadcast-box (good — the `any | null` device service fields preserve directional boundary at the cost of type safety). But the realtime module clearly **knows too much about broadcast-box concepts** — `DeviceConnectionMetadata` is defined in `src/types/realtime.types.ts:88-99`, there are dedicated `clientToDevice` / `deviceConnections` maps in the server (lines 121–128), and a 130-line "connection quality score" feature (`calculateConnectionScore`, `getDeviceConnectionsMetadata`, `checkStaleConnections`) is hard-coded for device connections even though every other concept in this module is record-agnostic. None of that should be in `@civicpress/realtime`.

**Integration with core.** Clean and consistent. `registerRealtimeServices` (`src/realtime-services.ts`) follows the documented Pattern 2 (singletons resolved from `ServiceContainer`); circular deps between `RealtimeServer` and `RoomManager` are broken with post-construction setters (`setRoomManager`, `setRecordManager`, `setDatabaseService`). No direct imports from `modules/api`, `modules/broadcast-box`, or `modules/ui`. The standalone bootstrap (`standalone-realtime.mjs`) is a reasonable adapter for "run realtime without API"; it does have to reach into `civicPress.container` (a private member) to wire broadcast-box dependencies post-init, which is a slight smell but is the explicit cost of late-binding.

**Service decomposition within the module.** Good: `RoomManager` (factory + lookup), `YjsRoom` (single-doc state), `SnapshotManager` + `SnapshotStorage` (persistence with database vs filesystem adapters), `PresenceManager` (presence state), `AwarenessManager` (Yjs awareness wrapper). Each is small (~100–400 lines), well-typed, and individually testable. The problem is concentrated in `RealtimeServer` itself, which acts as broker, router, message dispatcher, device-protocol translator, snapshot trigger, hook emitter, health monitor, and migration runner.

**Two routing strategies coexist.** `parseRoomId` (`src/auth.ts:355-380`) recognizes `records`, `record`, and `device` as room types; `RoomManager.registerDefaultRoomTypes` (`src/rooms/room-manager.ts:44-58`) registers both `record` and `records` factories pointing at the same `YjsRoom`. So `/realtime/records/abc` and `/realtime/record/abc` both work but create different rooms (they're keyed by full ID `records:abc` vs `record:abc`). `triggerRecordSnapshot` (`src/realtime-server.ts:3516`) papers over this by checking both formats. Either room-type plural-vs-singular should be normalized at the URL boundary, or one form should be removed.

## Roadmap Alignment

`docs/roadmap.md` does not mention collaborative editing or the realtime module at all. v0.2.x (claimed complete) talks about indexing and search; v0.3.x mentions "a richer Markdown editor" but not WebSocket sync; v0.4.x mentions "foundation for future collaborative editing" as a deliverable — the realtime module already exists and is claimed complete (`modules/realtime/IMPLEMENTATION-STATUS.md`, dated 2025-01-30), which **predates** v0.4.x. So either the roadmap is a phase behind reality (collab editing was a v0.4 deliverable, shipped early) or the realtime module is roadmap-orphaned scope that grew in service of broadcast-box (the device routing was added later, see CHANGELOG of broadcast-box commits in the same branch).

`docs/project-status.md` doesn't mention realtime at all in its "What's Working" section despite the module shipping. The most recent version of the file claims "1167+ tests passing" but the `CLAUDE.md` agent memory says "1291+ tests, 118 test files" and notes 111 realtime tests specifically — `project-status.md` is at least one cycle stale. The Transparency principle is mildly violated by an authoritative-looking status doc that doesn't match the code.

**Milestones unblocked by realtime (if findings addressed):** v0.4.x "Foundation for future collaborative editing" — already delivered; v0.5–0.8 "Pilot-ready CivicPress instance" — collab is a clear differentiator for clerk pilots if the offline-resilience and Markdown-write-back gaps are addressed.

**Milestones threatened:** v0.9 "Production Candidate / Security review" — the per-user limit dead code, the connection-count leak, and the unbounded snapshot blobs are all the kind of thing a real security audit would flag, and they're easy to fix now / hard to retrofit later. v1.0 "Cross-module stability" — the dual device-handling path means a breaking change to broadcast-box can be masked by the legacy fallback in realtime; that's the wrong direction for a stable v1.0.

## Notes / Open questions

- The realtime module presents itself as a self-contained collaborative-editing service, but in this branch it is co-evolving with broadcast-box. Two of the three biggest code blocks in `realtime-server.ts` (legacy device-message handler at line 1533, status/source/ack processing at lines 1782–2330) are broadcast-box specific. Phase 2 synthesis should consider whether to **extract a `@civicpress/realtime-broadcast-box` adapter** (or fully move device logic into `modules/broadcast-box/src/realtime/`) so that the realtime module can claim "generic Yjs WebSocket" honestly.
- The "collaborative edits don't write to Markdown" gap (finding realtime-003) is the single most manifesto-relevant issue here. The fix isn't obvious — naive "save every snapshot as Markdown" would conflict with TipTap's XmlFragment-as-source-of-truth model. A possible direction: a server-side Yjs→Markdown serializer (the inverse of TipTap's), invoked on snapshot. Worth a dedicated spike.
- Snapshot durability (realtime-005) needs a separate decision: are Yjs blobs treated as "ephemeral collaboration scratchpad" (in which case TTL them aggressively and don't rely on them for archival) or "civic record archive" (in which case version the format, sign them, size-limit them, and treat them as first-class storage)? The current code splits the difference, which is the worst of both.
- The `userConnections` map (`src/realtime-server.ts:117`) is also used by `getHealthStatus()` to report `perUser` count (line 3369). Because entries are never removed, this metric will grow monotonically and become misleading.
- The dual `record` / `records` room-type aliasing (`src/rooms/room-manager.ts:50-58`, papered over in `triggerRecordSnapshot` at line 3516) is the kind of "support both" decision that ages badly. Worth choosing one and migrating.
- `y-protocols` and `lib0` versions are pinned exactly (`lib0: 0.2.117`) while `yjs` is `^13.6.10` — minor-version drift in `yjs` could subtly break compatibility with the pinned encoder. Worth aligning the version-pinning strategy across the y-* deps.
- The standalone bootstrap (`standalone-realtime.mjs`) reaches into `civicPress.container` (private) to wire broadcast-box dependencies — a sign that the "register-then-initialize" lifecycle is the wrong shape for cross-module wiring. A clean fix would be a `civic.afterInitialize(callback)` hook.
- No tests were run during this audit (per scope). Findings are based on source reading. The `realtime.integration.test.ts` file does exercise reconnect/restore-after-disconnect, so finding realtime-003's "snapshot is the only persistence" model is at least partially covered by tests — but those tests do not assert that the Markdown record file is updated, only that the Yjs state replays.
