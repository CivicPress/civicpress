## Technical Quality

The module is roughly 29,300 lines of Vue/TS spread across 112 files in `app/`. Code is generally readable and uses Vue 3 Composition API consistently, with composables for cross-cutting concerns (`useApi`, `useAuth`, `useMarkdown`, `useAutosave`, `useRecordTypes`, `useRecordStatuses`, etc.). However, several "god-components" violate basic component-design hygiene: `RecordForm.vue` (1,310 lines), `FileBrowser.vue` (1,156 lines), `GeographyForm.vue` (1,104 lines), `RecordSidebar.vue` (935 lines), and `EditorHeader.vue` (600 lines). These mix presentation, API calls, state, and validation in single files and are difficult to test or modify safely.

Type safety is undermined by 208 explicit `: any` annotations and the response pattern `(await $civicApi(…)) as any` used pervasively (e.g., `pages/index.vue:147`, `pages/auth/register.vue:111`, `pages/records/[type]/[id]/index.vue:303–307`). The Pinia `records.ts` store also hard-codes the record-type and status unions (`'bylaw' | 'ordinance' | …`), directly contradicting the "config-driven types" claim in project-status.md. `useRecordTypes.ts` correctly fetches types from the API, but the icon and color maps inside it (`useRecordTypes.ts:110–139`) are still hard-coded English keys; any town that adds a `permit` or `agenda` record type via config gets a generic file icon and grey color.

Test coverage is essentially absent. Despite project-status.md claiming "UI 80+ tests passing, 85% coverage," the module contains exactly **one** test file with 32 cases for a single utility (`api-response.test.ts`). There is no `vitest.config.ts`, no `__tests__/` directories at the component or page level, and no e2e suite. None of the 112 components, pages, or composables are tested in isolation. Pre-commit hooks at repo root run vitest globally, but the UI module contributes virtually nothing to those numbers.

Error handling is centralized in `composables/useErrorHandler.ts` (good) but every title and fallback message inside it is hardcoded English (`'Connection Error'`, `'Validation Error'`, `'Unable to connect to the server. Please check your internet connection.'`). The composable also pulls `$toast` via `useNuxtApp()` but Nuxt UI Pro's toast API is accessed via `useToast()` elsewhere, so most of those toast calls silently no-op. Console logging is heavy: 112 `console.log/warn/error` calls remain in production code, including verbose auth-state debug logs in `stores/auth.ts:55–82,313,365` that leak session info to the browser console.

## Security (LIGHT)

**Critical XSS surface on the citizen-facing record page.** `useMarkdown.renderMarkdown` calls `marked.parse(source)` (`composables/useMarkdown.ts:140`) without DOMPurify or any sanitizer, and the result is fed to `v-html` in two places: `RecordPreview.vue:33` (clerk preview) and `pages/records/[type]/[id]/index.vue:675` (public record view). `marked` does not strip `<script>`, `<iframe>`, `onclick=`, `javascript:` URLs, or `<img onerror>` by default. A clerk (or anyone who can publish a record draft) can inject HTML that executes in every citizen's browser when they read that record — including stealing the JWT auth token from `localStorage` (`stores/auth.ts:46,114`) and the CSRF token from `localStorage` (`useCsrf.ts:28,45`). This is the central manifesto-Trust failure: civic records are exactly the surface that *must* be safe to display unauthenticated.

**Secondary script-injection surface in `app.vue`.** The root component fetches `/api/v1/info` on mount and, if the response contains `analytics.inject_head|inject_body_start|inject_body_end`, parses each as HTML and re-creates `<script>` elements with their attributes and inline content, appending them to `document.head` / `document.body` (`app/app.vue:62–134`). Any admin-controlled config that flows through this `/api/v1/info` endpoint becomes a code-execution lever. Even if the intent is "let admins paste a Plausible/GA snippet," there is no allow-list, no signed-source check, and no UI disclosure beyond the generic cookie banner. Combined with the XSS above, this means token theft is trivial if any of (a) the API server, (b) an admin account, or (c) a clerk record body is compromised.

**Tokens in `localStorage`.** Both the JWT and the CSRF token are stored in `localStorage` rather than `httpOnly` cookies. Any successful XSS reads them. The CSRF token's stated purpose ("for state-changing methods" `plugins/01-civicApi.ts:43`) does not protect against same-origin scripted requests when the attacker reads the bearer token in the same step. The route-guard middleware (`middleware/requireAuth.ts`, `middleware/requireAdmin.ts`) does the right thing on the client but enforces nothing if the API layer doesn't.

**Lower-severity issues.** `pages/records/drafts.vue:67` uses native `window.confirm()` for destructive draft deletion — not phishable but breaks i18n and a11y. `stores/auth.ts:260–284` exposes a `setMockUser()` action that grants `admin` role and unlimited permissions; no env gate or guard prevents this being invoked from devtools in production. CSP headers are not set anywhere in `nuxt.config.ts`. Surfaces that would benefit from a dedicated security session: (a) the record-rendering pipeline (markdown → HTML → v-html), (b) the `/api/v1/info` injection flow, (c) the auth-token storage model.

## AI-Generation Smells

Several pages bear strong Cursor/AI signatures. The clearest is the inconsistency between `pages/settings/setup.vue` and `pages/settings/configuration/index.vue` — both admin pages, both written in plausibly-correct Vue, both **completely bypass i18n** (hardcoded "Setup Wizard", "Configuration Status", "Refresh", "Validate Configuration", "Configuration Export feature coming soon", etc.) while every other settings page (`profile`, `diagnostics`, `storage`, `users/[username]/edit`) uses `t()` correctly. This is the classic "the model forgot the convention three turns ago" pattern, not a deliberate design choice. The Setup page even hardcodes the breadcrumb (`{ label: 'Home', to: '/' }`) while every other page uses `t('common.home')`. Similarly, `pages/records/[type]/[id]/index.vue` is mostly localized but contains stranded raw strings: `"Loading record..."` (line 572), `"Try Again"` (line 587), `"Record ID:"` (line 602), `"Record Not Found"` (line 905), `"Back to Records"` (line 910).

The `app.vue` analytics injector (lines 62–134) reads like AI-suggested code that no human security-reviewed: it manually re-parses `inject_head`, splits scripts from non-scripts, and re-creates `<script>` elements specifically to *re-execute* them. That's a deliberate choice no one would make without a threat model, yet there is no comment justifying it or warning future maintainers. The same file's `useHead` block declares both standard `<meta>` tags AND a separate inline `<script>` that re-declares `window.__NUXT_HEAD__` (line 43–58) for reasons that aren't explained — looks like patching around a Nuxt 4 SSR/SPA mode bug by sprinkling code until "it worked."

The error-handler composable (`useErrorHandler.ts`) has four nearly-identical handler functions — `handleApiError`, `handleNetworkError`, `handleValidationError`, `handleAuthError` — that each repeat the same ~30-line block of message extraction, log assembly, toast construction. The dispatch function `handleError` (line 256–275) just routes by HTTP status. This is the "AI loves five-method classes" smell; a single function with an options object would be half the code and easier to maintain. Also note that all four handler titles ("Error", "Connection Error", "Validation Error", "Authentication Error") are *not* `t()`-ed, defeating the purpose of i18n the rest of the app pays into.

`pages/records/[type]/[id]/edit.vue:163` uses permission-based access control with a comment "(consistent with API endpoints)" — yet `pages/records/new.vue:14–17` and `components/RecordList.vue:113–116` use role-based checks (`role === 'admin' || role === 'clerk'`) on the *same* operation (record create/edit). One of these was likely AI-pasted from an earlier convention and never reconciled. Similarly, `pages/records/drafts.vue` has a stranded `setMockUser` reference pattern and TODO scaffolding (`// Delete draft via API (we'll need to add this endpoint)` line 73) even though the endpoint clearly exists from the surrounding code.

## Architecture

The module mostly respects Nuxt 4 conventions: `app/`-rooted layout, pages under `app/pages/`, composables/components/stores in their conventional locations, plugins for cross-cutting concerns. Routing is convention-based and clean. The Pinia stores split is reasonable (`auth.ts`, `records.ts`, `app.ts`), and middleware (`requireAuth`, `requireAdmin`, `requireConfigManage`, `requireUsersManage`) maps to API permission keys.

Integration with `core/` and `modules/api/` is HTTP-only via `useNuxtApp().$civicApi`, which is the right boundary for a Nuxt SPA — no direct imports from the monorepo's TypeScript packages. The API base URL is correctly env-configurable (`API_BASE_URL`), and a Nitro dev proxy is wired up for `/api`. That said, the SPA-only choice (`ssr: false` in `nuxt.config.ts:4`) is at odds with both the manifesto's "citizen-discoverable" principle and basic SEO: bylaws and minutes are exactly the content that should be crawlable, archive-friendly, and viewable without JS. Switching to SSR or `nuxt generate` for the public record-reading paths would not be a small change — `useNuxtApp().$civicApi` is client-only, several composables guard with `process.client`, and the `app.vue` analytics-injection runs in `onMounted`. The architecture was designed for SPA and would need real rework to support a public-render mode.

`process.client` appears 33 times in the codebase — Nuxt 4 deprecates this in favor of `import.meta.client`. Not breaking yet, but a low-effort modernization debt. There is no offline-first strategy at all: no `@vite-pwa/nuxt`, no service worker registration in `public/manifest.webmanifest`, no IndexedDB layer for the autosave queue. For a tool whose flagship promise is "a town hall with a laptop and a USB key can run CivicPress … and remain functional in a storm, a blackout, or a service outage" (manifesto §3.2), the UI is the weakest link of the resilience story.

Component boundaries are mostly OK at the page level but break down inside the editor area. `RecordForm.vue` orchestrates EditorHeader, MarkdownEditor, EditorToolbar, PreviewPanel, RecordSidebar, and uses both `useAutosave` and `useRecordLock` — but it owns *all* the state for those children, including draft/published flow, autosave status, lock state, allowed transitions, and YAML reflection. The 1,310-line `<script setup>` is the single highest-risk file in the UI for unintended regressions. Decomposing it into a `useRecordEditor` composable + a slimmer `RecordForm` shell is a clear refactor target.

## Roadmap Alignment

The project-status.md claims for the UI module are partially aspirational. "Complete Internationalization (EN/FR)" overstates the reality: i18n locale files are well-built (1,007 EN keys, 1,005 FR keys, only ~10 keys missing in each direction), but several admin pages (Setup, Configurations index, parts of Users/[username]) hardcode English entirely, and `useErrorHandler.ts` produces only-English toast titles. "Complete read/write UI" is mostly true. "UI 80+ tests passing, 85% coverage" is **false** — one test file, 32 cases for one utility, zero component/page/composable coverage. The "Recently Completed" claim of a "Record Editor: Professional markdown editor with simplified button system" is accurate code-wise but the component is 1,310 lines and would not survive a refactor without regressions, given the zero test surface.

For the roadmap: v0.3.x ("Editor, Attachments, and Civic UX") and v0.5–0.8 ("Municipal Pilot Readiness") cannot ship safely until at minimum (a) markdown rendering is sanitized, (b) `@nuxt/ui-pro` either becomes verifiably free in production or is replaced, and (c) the UI gets meaningful test coverage. v0.9 ("Production Candidate / Accessibility audit") and v1.0 ("Stable Release") are blocked by the SSR/SPA decision: a citizen-facing civic platform that returns an empty `<div id="__nuxt">` to crawlers and JS-disabled users will not pass a WCAG/accessibility audit nor satisfy "if a citizen can't find it, it's not public." The flagship broadcast-box device page is not on this branch's working tree (only the modified file appears in `git status`); the WIP redesign on the `broadcast-box` branch is out of scope for this audit per instructions.
